# Долгий await в Telegram-обработчике блокирует бота для ВСЕХ чатов

**Rule:** PTB `Application` в проекте собран без `concurrent_updates=True`
(`agents/base_agent.py:1144-1149`, `Application.builder().token(...).post_init(...).build()`).
Апдейты обрабатываются строго последовательно — пока текущий обработчик не
вернётся, следующий апдейт (от ЛЮБОГО чата, не только текущего) не начнёт
обрабатываться. Любой прямой `await` долгой операции (сетевые синки с
retry/rate-limit sleep, несколько последовательных API-вызовов, ожидание
внешнего сервиса) прямо в хендлере команды/сообщения/callback — замораживает
ВЕСЬ бот: одобрение отзывов, другие команды, ИИ-чат в группах ждут в очереди,
пока не закончится операция одного пользователя.

**Why:** При реализации проактивного онбординга (`plans/2026-07-14-guided-
onboarding-analytics.md`) субагент по умолчанию написал `_send_finish` так,
что она `await`-ила полный синк (заказы → финотчёты → реклама → воронка,
может занимать 1-3 минуты из-за WB rate-limit `sleep(60)` на 429) прямо в
обработчике текстового сообщения онбординга. Баг найден оркестратором при
ревью диффа до коммита, не после — но легко было пропустить, если не знать
про `concurrent_updates=False`.

**How to apply:** Для любой операции длиннее одного быстрого API-вызова
(секунды, не десятки секунд/минуты), запускаемой из Telegram-хендлера:
1. НЕ `await` её напрямую в теле обработчика.
2. Использовать `asyncio.create_task(self._some_background_method(...))` —
   паттерн уже применялся в проекте до этой фичи (ДРР через Питера,
   `agents/max.py:6809`, `_asyncio.create_task(peter.run_drr_for_chat(...))`).
3. Обработчик должен вернуться сразу после короткого подтверждающего
   сообщения ("собираю данные, отвечу здесь через минуту...").
4. Фоновый метод обязан сам ловить исключения (try/except с логированием +
   уведомлением пользователя) — `create_task` без `await`/done-callback
   молча проглотит необработанное исключение (попадёт только в лог asyncio
   "Task exception was never retrieved").

Если в будущем понадобится настоящая параллельная обработка апдейтов —
это отдельное архитектурное решение (`concurrent_updates=True` в
`Application.builder()`), не делать явочным порядком в одной фиче.
