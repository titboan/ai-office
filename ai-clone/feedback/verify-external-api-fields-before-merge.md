Если код сопоставляет результат внешнего API со списком захардкоженных строк (имена сервисов, типы операций, enum-значения) — нельзя мерджить эту логику только на основе документации/предположений. Нужен хотя бы один прогон на реальных данных, подтверждающий совпадение.

**Почему:** В сессии 2026-06-15 был смерджен `get_fin_adv_spend` (`tools/marketplace.py`) — функция, которая ищет рекламные расходы Ozon (Premium, бренд, CPC, CPO) по списку угаданных имён `_MARKETING_SERVICES` (`SellerClicksMarketing`, `SellerPremiumSubscription` и т.п.). Документация Ozon API недоступна для автоматической проверки (блокирует фетч), поэтому имена были придуманы по аналогии. На проде ни одно угаданное имя не совпало ни разу — реальные `services[]` содержат только логистику. Фича целый день показывала заниженное ДРР, прежде чем баг заметили по разнице с реальным отчётом Ozon.

**Как применять:**
- Если нет доступа к документации/тестовому стенду внешнего API — разворачивать новую фичу сопоставления полей в **диагностическом режиме**: логировать факт (что реально пришло), но не использовать результат в расчётах/алертах/ДРР, пока не подтверждено совпадение хотя бы одним реальным прогоном в проде или на тестовых данных.
- Только после того, как пользователь подтвердил по логам, что угаданные имена совпали с реальностью — включать значения в расчёт.
- Это отдельная категория от "выдуманных данных" — здесь дело не в честности, а в том, что строковое совпадение с внешним API нельзя проверить статическим анализом, только эмпирически.
