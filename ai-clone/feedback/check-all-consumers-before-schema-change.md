Когда меняешь формат данных, который отдаёт общий метод (например, `agents/peter.py::_collect_data`),
нужно сразу искать всех потребителей этого поля — не только то место в той же функции, где видна
прямая правка (текстовый промпт), но и любые внешние API-эндпоинты, фронтенды, дашборды, которые
на него завязаны.

**Почему:** изменил структуру `net_margin` (с построчного `marketplace/product_id/net_margin_pct`
на одну строку на товар с колонками `_wb/_ozon/_total`) с оглядкой только на промпт Питера. Не
проверил, что `_collect_data` — общий источник и для чата, и для `/api/dashboard`
(`main.py::_handle_dashboard`). Фронтенд дашборда (`MarginChart.tsx`, `NetMarginTable.tsx`, тип
`NetMarginRow`) остался на старой схеме, React падал с TypeError при обращении к несуществующим
полям — без error boundary это давало чёрный экран в проде. Деплоя без команды не было,
`git add` поимённо соблюдался — формально правила репо не нарушены, но смысл «не ломать рабочую
функциональность» был нарушен молча, пользователь обнаружил баг сам после пуша.

**Как применять:** перед пушем любого изменения формата данных, отдаваемого наружу (HTTP API,
файл, общий dict/JSON, любая структура, видимая за пределами текущей функции) — обязательно
`grep`/поиск по имени поля во всём репо (включая `dashboard/`, любые фронтенды, другие агенты),
а не только в файле, где правишь код. Если есть TS/JS потребитель — прогнать `tsc --noEmit`
после правки соответствующего интерфейса, это ловит порядок несовпадений до деплоя.

См. также [[verify-external-api-fields-before-merge]], [[verify-join-keys-before-sql-merge]].
