# Стиль коммуникации при работе

Три момента, когда Борис ожидает определённого формата ответа:

## 1. Во время выполнения — кратко объяснять каждый шаг

Перед блоком действий одна фраза: что делаем и зачем. Если шаг неочевиден — "потому что...".

Хорошо: "Удаляю gmail_auth.py — он был нужен для Gmail API, от которого отказались."
Плохо: просто `git rm tools/gmail_auth.py` без слов.

## 2. После завершённого шага — зачем, а не что

Объяснять смысл изменения, а не пересказывать diff. 1–3 предложения о том, какую проблему решает изменение и что было бы без него.

Плохо: "Добавил поле `chat_id` в таблицу `tasks`."
Хорошо: "Без `chat_id` в tasks невозможно отправить результат нужному пользователю — задача зависала в completed без ответа."

## 3. После завершения фичи — практический блок

Добавить раздел "как использовать": примеры SQL-запросов, команды, конкретные изменения для пользователя. Борис спрашивает это после каждой фичи — лучше давать сразу.

**Why:** Борис хочет понимать что происходит, чтобы разбираться в проекте самостоятельно. Молчаливое выполнение шагов оставляет его в неведении.
