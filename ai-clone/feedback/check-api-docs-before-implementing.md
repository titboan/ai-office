Перед написанием или изменением кода, который вызывает внешний API (WB, Ozon, Telegram и др.) — обязательно найти и прочитать актуальную документацию для конкретного эндпоинта. Нельзя строить реализацию на догадках, даже если паттерн «похож» на другой эндпоинт того же API.

**Почему:** В сессии 2026-06-28 несколько дней не работал ответ на вопросы покупателей WB. Причина — два неверных предположения одновременно:
1. `"text"` клался на верхний уровень тела запроса вместо вложенного объекта `"answer": {"text": "..."}`
2. `"state"` имел значение `"wbGoodsQaStatePublished"` вместо правильного `"wbRu"`

WB API возвращал 200/204 (успех) но тихо игнорировал запрос. Это привело к:
- Бесконечному циклу переотправки уведомлений в группу
- Нескольким неверным итерациям (сначала изменили state, потом endpoint на POST, потом вернули PATCH) — каждый раз на основе догадки, а не документации

Правильный формат был найден только через поиск в OpenAPI-спецификации `eslazarev/wildberries-sdk`.

**Как применять:**
- Перед любым новым API-вызовом: найти документацию (OpenAPI spec, swagger, SDK) и прочитать точный формат
- Если документация недоступна напрямую (WB блокирует fetch): искать через `WebSearch` + открытые GitHub-репозитории с OpenAPI-спецификациями
- Нельзя делать реализацию «по аналогии» с другим эндпоинтом того же API — форматы часто отличаются
- Если документация не найдена вообще — явно сказать пользователю ДО написания кода: «не могу найти документацию, реализую диагностический режим»
- После нахождения документации — зафиксировать в SKILL.md и в комментарии к функции ссылку на источник

**Для WB Questions API (найдено):** PATCH `feedbacks-api.wildberries.ru/api/v1/questions`, body: `{"id": "...", "answer": {"text": "..."}, "state": "wbRu"}`. Источник: `eslazarev/wildberries-sdk specs/09-communications.yaml`.
