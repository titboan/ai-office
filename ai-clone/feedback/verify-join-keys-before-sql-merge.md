Перед тем как писать или менять SQL-джойн между таблицами по `product_id` (или любому другому общему полю), нужно на реальных данных подтвердить, что обе стороны действительно хранят значения одной семантики — а не предполагать единообразие по имени колонки.

**Почему:** В сессии 2026-06-16 правил формулу ROAS в `agents/peter.py`, сменив источник `buyouts` с `marketplace_orders` на `marketplace_sales`, и джойнил по `product_id = product_id` напрямую. Не проверил, что для Ozon `marketplace_stocks`/`marketplace_sales` хранят `offer_id` (буквенный), а `marketplace_orders`/`product_adv_stats` хранят `sku` (числовой) — для WB везде один и тот же `wb_article`, поэтому раньше эта асимметрия маскировалась. В итоге фикс сам сломал ROAS для всех Ozon-товаров (всегда 0) и независимо обнажил старый баг в `stock_velocity` (Ozon-остатки пропадали из дашборда — артикул КБ50 был виден только на WB). Баг нашли только когда пользователь прислал конкретный пример несовпадения.

Промежуточная попытка фикса (джойн через `product_mapping.id` без разбивки по marketplace) сама оказалась багом — она задвоила скорость продаж WB и Ozon в одно число, что заметили только перепроверкой на реальных данных после "исправления".

**Как применять:**
- Перед джойном по `product_id` между двумя таблицами — одним SQL-запросом проверить на реальных строках, что формат идентификатора совпадает (особенно если задействован Ozon: `offer_id` ≠ `sku`, см. `product_mapping.ozon_offer_id`/`ozon_sku`).
- После написания джойна — обязательно прогнать на реальных данных и руками сверить 1-2 строки, а не доверять только `py_compile`/синтаксической валидности.
- Если джойн объединяет данные по нескольким marketplace — проверить, что результат не "перетекает" между площадками (то есть скорость продаж/метрика WB не подмешивается в Ozon-строку и обратно).
- Связано с [[verify-external-api-fields-before-merge]] — там про внешний API, здесь про собственную схему БД, но принцип общий: совпадение по имени поля/семантике нельзя проверить чтением кода, только данными.

**WB-специфика (добавлено 2026-06-28):** `WBClient.get_financial_report()` хранит в `marketplace_financial_report.product_id` либо `sa_name` (текстовый артикул вроде "КБ50"), либо числовой `nm_id` — в зависимости от того, что вернул WB API (если `sa_name` пустой, пишется `nm_id`). Поэтому джойн с `product_mapping` должен проверять **оба варианта**:
```sql
ON (f.marketplace = 'wb' AND (
        LOWER(REPLACE(m.wb_article, ',', '.')) = LOWER(REPLACE(f.product_id, ',', '.'))
        OR m.wb_nm_id::text = f.product_id
    ))
```
Если написать только по `wb_article` — все строки где WB API отдал `nm_id` дадут `qty_wb = 0` и "—" в дашборде без явной ошибки.
