При диагностике бага "Питер не видит данные" (2026-06-24) выявлено несколько структурных проблем в `agents/peter.py`:

**Why:** Все исключения в блоках сбора данных handle_task уходят только в logger.warning. Claude при этом получает prompt без данных и отвечает "нет инструментов/данных" — непонятная ошибка для пользователя.

**How to apply:** При любом исключении в сборе данных Claude должен получать явный текст об ошибке в data_str, а не пустую строку. Исправлено для основного data-блока (2026-06-24). SEO-fallback (строка 839) — та же проблема, ещё не исправлена.

## Оставшиеся баги в handle_task (не исправлены)

### Баг 1 — SEO fallback блокирует сбор данных
Если SEO-ключи совпали, но `_collect_seo_audit_data` упала с исключением:
```python
except Exception as e:
    prompt = f"Аналитическая задача от {from_agent}: {task}"  # prompt непустой!
```
Дальше `if not prompt and chat_id and any(supply_kw)` → False.
Затем `if chat_id and not prompt:` → False.
→ data_str остаётся пустым, основные данные не собираются. Claude отвечает без цифр.

**Фикс**: в SEO except-блоке ставить `prompt = ""` вместо базового текста, чтобы дальнейшие блоки могли собрать данные.

### Баг 2 — run_task без _current_chat_id (Марта → Питер напрямую)
В двух местах marta.py Марта вызывает `agent.run_task(subtask, from_agent="Марты")` напрямую (строки 702 и 1100), не устанавливая `agent._current_chat_id`. Питер получает `chat_id = 0` → данные не собираются.

Строка 702 — fallback когда task_queue недоступен.
Строка 1100 — делегирование из handle_task Марты.

**Фикс**: перед вызовом `agent.run_task()` устанавливать `agent._current_chat_id = chat_id`, после — сбрасывать.

### Баг 3 — handle_task игнорирует период из текста задачи
`_collect_data(chat_id, days=14)` — всегда 14 дней. Если пользователь написал "за 30 дней" — Питер всё равно берёт 14-дневные данные и отвечает на основе них, не предупреждая.

**Фикс**: парсить `days` из текста задачи простым re-поиском ("за (\d+) дней"), дефолт — 14.
