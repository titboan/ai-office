# Рефлексия: баг в таблицах — конфликт форматов в /funnel

**Дата:** 2026-06-15  
**Сессия:** диагностика и фикс бага форматирования таблиц

---

## 1. Какая задача была

Пользователь показал скриншот Telegram с отчётом Питера — таблицы видны, попросил проверить все и найти баги.

## 2. Как решали

1. Прочитал `utils/tg_rich.py` и `utils/tg_format.py` — понял архитектуру Rich Markdown и HTML fallback.
2. Прочитал `agents/peter.py` полностью — все 5 команд Питера.
3. Сравнил system prompt, user prompt и send path для каждой команды.
4. Проверил другие агенты на похожие паттерны (grep по `parse_mode`, `send_rich`, системным промптам).

## 3. Решили — да

Найден и исправлен один баг: в `cmd_funnel` user prompt содержал HTML-примеры (`<b>`, `<code>`, `<blockquote>`) при системном промпте Rich Markdown и send path через `sendRichMessage`. Клод следовал HTML-примеру из user prompt, а не системному промпту → теги рендерились как текст.

Фикс: конвертировал HTML-примеры в user prompt на Rich Markdown (`**текст**`, `` `код` ``, `> цитата`).

## 4. Что можно было лучше

- Баг появился потому что `/funnel` писался раньше (или параллельно) с миграцией на Rich Markdown, и user prompt не обновили вместе с system prompt. При следующей миграции форматов — обновлять не только системные промпты, но и все user prompt с inline-примерами формата.
- Рефлексия предыдущей сессии (`2026-06-14_markdown-v2-migration.md`) уже описывала правило "grep по всем `parse_mode=`" — но user prompt нужно добавить в scope этого grep.

## 5. Что узнал нового о проекте

- Питер единственный агент с несколькими форматами одновременно: `/report` и `/funnel` — Rich Markdown, `/audit` и `/drr` — HTML. Это осознанно, но создаёт риск рассинхрона при будущих изменениях.
- В HTML-fallback таблицы удаляются (`re.sub(r"^\|.*\|[ \t]*$", "", text)`). Пользователь не увидит таблицу если `sendRichMessage` недоступен — но это приемлемо.

## 6. Нарушений правил не было

Агент не нарушал правила молча. Фикс — минимальный, в одном месте, без рефакторинга.
