# Рефлексия: Cost Tracking

Дата: 2026-06-13

---

## 1. Какая задача была

ROADMAP Phase 6: добавить `estimated_cost` и `latency_ms` в таблицу `tasks`.
Сейчас токены только логируются в DEBUG — стоимость не считается, латентность не хранится.

---

## 2. Как решали

- `db.py`: два `ALTER TABLE IF NOT EXISTS` — добавили `estimated_cost NUMERIC(10,6)` и `latency_ms INT`
- `task_queue.py`: функция `update_task_cost(task_id, cost_usd, latency_ms)` — fire-and-forget, ошибки подавляются через `logger.debug`
- `agents/base_agent.py`:
  - `_COST_PER_1M` — словарь цен по трём моделям (sonnet, opus, haiku)
  - `_calc_cost(model, input, output)` — одна строка математики
  - `think()` — после каждого вызова Claude накапливает токены в `self._task_tokens` (если атрибут существует)
  - `_worker_loop()` — сбрасывает `_task_tokens` перед `handle_task()`, фиксирует `time.monotonic()`, после `mark_completed` вычисляет латентность и вызывает `update_task_cost`

Синтаксис-проверка через `ast.parse` с `encoding='utf-8'` — всё ОК.

---

## 3. Решили — да / нет / частично

**Да, полностью.**

---

## 4. Что можно было лучше

- Цены моделей (`_COST_PER_1M`) вынесены прямо в `base_agent.py`. Правильнее было бы положить их в `config.py` в секцию `CONSTANTS` — чтобы менять в одном месте без правки кода. Сознательный компромисс ради простоты.
- Не учитывается стоимость вызовов через `claude-opus-4-8` в `_plan_chain()` у Марты — там создаётся отдельный `self.claude.messages.create` напрямую, не через `think()`. Стоимость цепочек будет занижена на стоимость планировщика. Это приемлемо для первой итерации.
- Latency меряется от `mark_running` до `mark_completed`, не включая время в очереди — это правильное решение, latency = время работы агента.

---

## 5. Что узнал нового о проекте или пользователе

- В `_worker_loop` есть прямые вызовы `claude.messages.create` вне `think()` (в Марте, Кевине) — Cost Tracking через `_task_tokens` их не поймает. Для полного трекинга нужно либо оборачивать все вызовы в `think()`, либо добавить отдельный перехватчик.
- Паттерн fire-and-forget с `logger.debug` (а не `logger.error`) — правильный для вспомогательных операций, которые не должны ронять основной поток.
