# 2026-07-18 — Потеря и искажение данных при синке Макса (WB/Ozon)

## Задача

Продолжение сессии после мержа предыдущего плана (критичные баги цены/ставок). Пользователь
попросил "продолжи" — выбрал следующий по приоритету открытый план из аудита 2026-07-18:
`plans/2026-07-18-max-sync-data-loss.md` (9 багов синка данных WB/Ozon), т.к. из оставшихся
он про потерю/искажение реальных бизнес-данных (отзывы, KPI, остатки, заказы), а не про UX.

## Как решали

Все 9 фаз по очереди, каждая — отдельным вызовом `implementer` с точным техзаданием (готовый
код, конкретные строки), сам проверял diff перед коммитом:

1. Ретрай зависших `pending_approval` отзывов (по аналогии с уже существующим ретраем вопросов) —
   `rv_notified:{mp}:{review_id}` в Redis, TTL 2ч.
2. Пагинация WB отзывов/вопросов и Ozon отзывов (были ограничены одной страницей).
3. `last_checked_at` магазина теперь реально проставляется после успешного синка (раньше только
   сбрасывался в NULL для отладки, никогда не писался).
4. `shop_id` в `shop_kpi_snapshots`/`marketplace_in_transit` — та же миграция, что раньше уже
   делали для `marketplace_stocks`/`financial_report`/`fin_adv`. `sync_shop_kpi` возвращает
   результат по `shop_id`, а не по `marketplace`.
5. WB реклама: retry текущего батча на 429 (раньше `continue` переходил на следующий батч,
   теряя статистику по 50 кампаниям).
6. Логирование обрыва пагинации Ozon на `offset >= 2000`.
7. CSV Ozon: fallback на `cp1251`, если `utf-8-sig` дал мусор; логирование пропущенных батчей
   с именем файла.
8. Перевод 5 методов `WBClient` на общий `_request()` (ретраит 429/500/502/503) — раньше свои
   ручные циклы ретраили только 429.
9. `check_connection` (WB/Ozon) различает `"invalid_token"` и `"unavailable"` — онбординг
   подключения магазина больше не путает временный сбой маркетплейса с неверным токеном.

## Решили — да / нет / частично

Да, план полностью завершён (9/9 фаз), статус `завершён`, 9 коммитов, ветка
`claude/priority-plan-uwo5oz` запушена. PR не создавал — пользователь не просил явно в этой части
сессии (в начале сессии просил "мержи" для предыдущего плана и это было сделано отдельным PR #32).

## Что можно было лучше

- Фаза 4 (`shop_id` в KPI) — implementer сознательно оставил READ-запрос плана поставки
  (`marketplace_in_transit` без фильтра по `shop_id`) и дашборд-контракт `shop_kpi` (по-прежнему
  индексированный по `marketplace`) нетронутыми, как и просило задание — но это значит, что
  полная защита от "два магазина одного маркетплейса" пока однобокая: запись честная, часть
  чтения — нет. Не баг (для текущего single-shop клиента), но стоит держать в уме как известное
  ограничение, если DoggyDog когда-нибудь подключит второй магазин Ozon.
- Не тестировал в реальном Telegram/дашборде (нет доступа к живым WB/Ozon токенам в этой среде).

## Что узнал нового о проекте

- В файле `tools/marketplace.py` исторически сложилось два стиля HTTP-запросов: общий `_request()`
  хелпер (ретраит 429/5xx, используется в новых методах) и старые ручные `for attempt in range(2)`
  циклы (ретраят только 429, привнесены до появления `_request()`). Докстринг модуля обещает
  поведение `_request()` для ВСЕХ методов, но по факту это выполнялось не везде — стоит иметь в
  виду при следующих правках сетевого кода в этом файле: сверяться с докстрингом, а не с
  соседним кодом.
- `sync_prices()`/паттерн "не писать оптимистично, довериться следующему честному синку" (уже
  использованный в прошлой сессии для цен) — тот же принцип применим и к `last_checked_at`:
  проставлять реальную метку времени только после подтверждённого успеха, а не заранее.

## Нарушил ли агент правило молча?

Нет явных нарушений. В Фазе 1 предыдущего плана (не этого) была самостоятельная правка
`main.py` вне заявленного скоупа implementer'а — здесь такого не потребовалось, все 9 фаз этого
плана обошлись без выхода за пределы явно заданного скоупа.
