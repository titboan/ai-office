## Задача

Пользователь прислал скриншот из приложения Ozon: несколько вопросов покупателей
(с августа и ноября 2025) висят без ответа со статусом «Нужен ответ». Жалоба:
«с Озона не приходят вопросы».

## Как решали

Код для Ozon Q&A (`tools/marketplace.py::OzonClient.get_questions/answer_question`)
был написан по аналогии с другими Ozon-эндпоинтами (`page`/`page_size`, поля
`question_text`/`product_name`) без проверки реальной документации — тот же
класс ошибки, что уже случался с WB questions API (см.
`ai-clone/feedback/check-api-docs-before-implementing.md`). Официальная документация
Ozon недоступна для прямого fetch (403), поэтому нашли точный контракт через
комьюнити-SDK с полным покрытием API — `github.com/salacoste/ozon-daytona-seller-api`
(`src/types/{requests,responses}/questions-answers.ts`).

Обнаружены и исправлены три независимых дефекта:
1. `get_questions` слал `{"page", "page_size", "status": "not_answered"}` —
   у Ozon курсорная пагинация (`last_id`) и `filter.status` с enum
   `NEW/VIEWED/PROCESSED/UNPROCESSED/ALL`. Полей `page`/`page_size` в API нет.
2. Текст вопроса читался из несуществующих полей `question_text`/`question` —
   реальное поле называется `text`. Даже если бы пагинация работала, текст
   вопроса всегда приходил бы пустым.
3. `answer_question` не отправлял обязательное поле `sku` — Ozon отклоняет
   такой запрос, поэтому одобренный в Telegram ответ физически не мог дойти
   до Ozon.

Также подтверждено: Q&A API требует подписку Premium Plus (как и отзывы,
уже помеченные в SKILL.md как недоступные). Раз в приложении Ozon вопросы
видны, а через API — не факт, что подписка на Q&A вообще активна на этом
магазине; это отдельный вопрос к пользователю, не баг кода.

Изменения:
- `tools/marketplace.py` — исправлен формат запроса `get_questions`, маппинг
  полей ответа, добавлено обязательное `sku` в `answer_question` (с валидацией).
- `agents/max.py` — оба места вызова `answer_question` теперь достают
  `product_id` (sku) из БД (`marketplace_questions`) и передают его дальше.
- `.claude/skills/max-api/SKILL.md` — задокументирован подтверждённый контракт
  Ozon Q&A API и источник.
- `tests/test_ozon_questions.py` — regression-тесты на все три дефекта
  (формат запроса, извлечение текста, обязательность sku).

## Решили — да / нет / частично

Частично. Код исправлен и покрыт тестами, но не проверено на реальном аккаунте
Ozon (нет доступа к продовым секретам из этой сессии) — не подтверждено,
доступен ли Q&A API вообще на текущей подписке. Нужно проверить после деплоя:
если Premium Plus не подключён, `get_questions` по-прежнему будет получать
PermissionDenied и это нормально (как с отзывами), а не повторный баг.

## Что можно было лучше

Сразу проверять формат внешних API по документации/SDK перед тем как писать
клиент — этот баг был предотвратим, если бы предыдущий инцидент с WB questions
(см. feedback-файл) был применён как чек-лист и к Ozon-части того же плана.

## Что узнал нового

- Ozon Questions & Answers API — отдельная, менее задокументированная категория
  методов (`/v1/question/*`), требует Premium Plus, использует курсорную
  пагинацию `last_id` везде, где в других эндпоинтах Ozon используется
  `limit`/`offset`.
- Официальная документация Ozon блокирует прямой fetch (403); community SDK
  с полным покрытием API (daytona) — рабочий источник для сверки контрактов,
  когда официальные доки недоступны.

## Нарушил ли агент правило молча

Нет. Внешний API-вызов был исправлен только после сверки с документацией
(через community SDK), не по аналогии. Prod-доступ (Railway CLI, `.env`,
живые кредлы Ozon) не использовался и не запрашивался — все выводы сделаны
по чтению кода и внешней документации.
