# 2026-06-24 — Питер не видит данные из БД

## Задача
Борис прислал скриншот: Питер отвечает на "за 30 дней, обе площадки" фразой "у меня нет доступа к инструментам или данным — ты отправил только текст задачи без самих цифр".

## Как решали
Читал `handle_task` в `agents/peter.py` (~811-888). Нашёл два бага:

**Баг 1 — структурный (else: prompt = ""):**
Блок supply-ключевых слов имел `else: prompt = ""`. Этот `else` выполнялся всегда, когда supply-ключи не совпадали — сбрасывал `prompt` даже если SEO-блок его уже заполнил выше.

**Баг 2 — главная причина симптома:**
Если `_collect_data` или `_collect_advanced_data` падали с исключением, `data_str` оставался пустой строкой. Claude получал только текст задачи без данных и отвечал "нет инструментов". Исключение молча уходило в `logger.warning`.

## Решение
1. Удалил `else: prompt = ""` у блока supply (не нужен: внутри `if` и так ничего не меняется при пустом условии)
2. При исключении в data-сборе — `data_str` теперь содержит описание ошибки, которое Claude передаёт пользователю

## Решили: да / частично
Оба бага исправлены. Но корневая причина исключения неизвестна — видна только в логах Railway. Возможно: нестабильный pool, пустые данные в БД, несовпадение chat_id.

## Что можно было лучше
- Сразу посмотреть логи Railway (там была бы строчка `[Питер] handle_task: ошибка сбора данных: ...`)
- Баг 1 существовал давно: структура if/else в handle_task написана небрежно

## Что узнал нового
- `handle_task` Питера жёстко хардкодит 14 дней — игнорирует "за 30 дней" из текста задачи
- При прямом вызове `agent.run_task()` от Марты (fallback и delegation) `_current_chat_id` НЕ устанавливается → chat_id = 0 → Питер тоже не получит данные на этом пути
- SEO-блок при исключении ставит `prompt = "basic task"`, после чего `if chat_id and not prompt:` False → данные не собираются тоже (третий скрытый баг)
