# Рефлексия: автосинк дашборда, формула ROAS, и баг с Ozon offer_id/sku

**Дата:** 2026-06-16
**Сессия:** "на дашборде нет актуальных данных за сегодня" → автосинк → ROAS → редизайн остатков → найден сквозной баг джойнов по Ozon ID

---

## 1. Какая задача была

Началось с жалобы: "на дашборде нет актуальных данных за сегодня". По ходу сессии добавилось ещё четыре связанных запроса от пользователя: автоматическое расписание синхронизации (с интервалом 1 час), пересчёт формулы ROAS (выкупы/расход вместо заказов/расход), редизайн блока остатков без дублей по площадкам, и наконец — разбор конкретного кейса (артикул КБ50 виден на Ozon, но не на дашборде), который вскрыл отдельный системный баг.

## 2. Как решали

1. Root-cause "нет данных за сегодня" нашёл через прямой Postgres-запрос: `sync_marketplace_data()` вызывался только вручную (кнопка/команда), фонового цикла для заказов/продаж/остатков не было — в отличие от рекламы (03:00 UTC) и финотчёта (вс 01:30). Добавил `_scheduled_orders_sync_loop()` в `main.py` по аналогии с существующими циклами, по запросу пользователя — интервал 1 час вместо изначально предложенных 4.
2. Поправил формулу ROAS в `agents/peter.py`: источник сменил с `marketplace_orders` (заказы) на `marketplace_sales WHERE is_return=FALSE` (реальные выкупы) — по явному требованию пользователя.
3. Переделал `StockTable.tsx` — с плоской таблицы (дублирующиеся строки на WB+Ozon) на карточный список, группировка по имени товара на фронте.
4. Пользователь прислал конкретный кейс: КБ50 есть на Ozon, но на дашборде показан только WB. Раскопка показала: это НЕ баг группировки на фронте (как я предположил сначала), а баг в самом SQL на бэкенде — `marketplace_stocks`/`marketplace_sales` хранят для Ozon `offer_id`, а `marketplace_orders`/`product_adv_stats` хранят `sku`. Прямой join `product_id = product_id` никогда не совпадал для Ozon → `daily_orders` всегда 0 → `days_left=999` → строка вылетала из `ORDER BY days_left ASC LIMIT 15`. Тот же баг означал, что ROAS для ВСЕХ Ozon-товаров после моего фикса в п.2 стал 0 (выкупы не находились).
5. Первая попытка фикса (джойн через `product_mapping.id` без учёта marketplace) сама была багом — она задвоила скорость продаж: WB и Ozon продажи объединялись в одно число и применялись к обеим строкам одинаково (проверил на реальных данных, увидел одинаковый `daily_orders` у WB и Ozon строк — стало понятно, что неправильно). Переписал на marketplace-aware трансляцию (`CASE WHEN marketplace='ozon' THEN COALESCE(mapping.., product_id) ELSE product_id END`), сохранив раздельную агрегацию по площадкам.
6. Проверил оба фикса на реальных данных в Postgres (через `asyncpg`, не `psql` — не установлен локально) до коммита.

## 3. Решили — да

Все четыре явных запроса пользователя выполнены: автосинк добавлен и запущен (по коммиту), ROAS пересчитан, остатки редизайнены, и вдобавок найден и исправлен более глубокий баг (Ozon offer_id/sku рассинхрон), который пользователь не формулировал явно как задачу — он просто дал один конкретный пример ("КБ50 есть на Ozon, но не показывается"), а я довёл до корня.

## 4. Что можно было лучше

- **Не проверил данные перед тем как сделать первый ROAS-фикс.** Когда менял формулу ROAS с заказов на выкупы (п.2), не проверил, что `product_adv_stats` (откуда я джойнил) и `marketplace_sales` используют РАЗНЫЕ идентификаторы для Ozon. Из-за этого свежий фикс сам по себе сломал ROAS для всех Ozon-товаров (всегда 0) — баг просуществовал в `main` несколько коммитов, пока пользователь не прислал конкретный пример с КБ50. Это тот же паттерн, что описан в `retrospectives/2026-06-16_ozon-drr-неизвестные-списания.md`: менять формулу/джойн на разнородных по семантике полях без эмпирической проверки на реальных данных.
- **Промежуточная попытка фикса сама была багом** (объединение WB+Ozon в одну скорость продаж) — поймал её только потому, что explicitly проверил на реальных данных после фикса, увидел одинаковые числа у разных marketplace-строк и насторожился. Если бы не было привычки проверять на реальных данных после каждого SQL-фикса (а не верить, что синтаксис == корректность) — задвоение прошло бы в прод.
- Имеет смысл задокументировать соответствие `offer_id` vs `sku` по таблицам в `.claude/skills/db-schema/SKILL.md`, чтобы при следующем джойне через Ozon product_id это не пришлось заново раскапывать через прямые SQL-запросы.

## 5. Что узнал нового о проекте

- В рамках Ozon у одного товара де-факто ДВА разных "product_id" в зависимости от таблицы:
  - `offer_id` (буквенный, как у WB-артикула) — хранится в `marketplace_stocks`, `marketplace_sales`
  - `sku` (числовой) — хранится в `marketplace_orders`, `product_adv_stats`
  - `product_mapping` содержит оба поля (`ozon_offer_id`, `ozon_sku`) и может служить мостом между ними — но мостом нужно пользоваться ЯВНО и per-marketplace, не просто джойнить таблицы между собой по `product_id`.
- Для WB такой проблемы нет — везде используется один и тот же `wb_article`.
- Любой будущий SQL-джойн между таблицами, где участвует Ozon `product_id`, должен сначала проверить, какое именно поле (`offer_id` или `sku`) использует каждая из сторон, а не предполагать единообразие.

## 6. Нарушил ли агент правило молча?

Да — по сути повторил паттерн из retrospectives/2026-06-16_ozon-drr: смержил формулу с разнородным джойном по внешнему/составному ID без проверки на реальных данных ДО первого коммита. В прошлый раз для такого случая уже была мысль зафиксировать правило в `ai-clone/feedback/`, но более общего правила про "проверять semantics совпадения ID между таблицами при джойне" там пока нет — стоит дополнить.
