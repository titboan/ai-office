# Рефлексия: неизвестные списания Ozon и попытка починить ДРР

**Дата:** 2026-06-16
**Сессия:** разбор ntfy-алерта "новый тип рекламных расходов" → ревизия фичи get_fin_adv_spend

---

## 1. Какая задача была

Прилетел автоматический ntfy-алерт: в финотчёте Ozon нашлись 8 "неизвестных" списаний на сотни тысяч рублей, не учтённых в ДРР. Алерт прямо просил добавить их в `_MARKETING_SERVICES`. Пользователь сразу усомнился в этом — и оказался прав.

## 2. Как решали

1. Разобрал по смыслу все 8 названий (DirectFlowLogistic, LastMileCourier, BrandCommission и т.д.) — все оказались логистикой/комиссиями/эквайрингом, а не рекламой. Добавление их в ДРР задвоило бы расход (логистика уже учтена в NET-марже через `payout`).
2. Завёл `_NON_MARKETING_SERVICES`, чтобы не задваивать и не спамить алертом — закоммитил и смерджил (PR #9).
3. Пользователь прислал скрин реального финотчёта Ozon (раздел "Продвижение и реклама", 207 695 ₽: Premium 9990, бренд 9723, CPO 71242, CPC 116740) и показал, что дашборд Марты всё равно показывает 103 746,81 ₽ — расхождение почти в 2 раза.
4. Раскопал, что НАКАНУНЕ (за несколько часов до этой сессии) в `main` уже был смерджен другой фикс — `get_fin_adv_spend` с захардкоженным `_MARKETING_SERVICES` (угаданные имена типа `SellerClicksMarketing`, `SellerPremiumSubscription`). Попросил пользователя синхронизировать вручную и прислать сырой лог.
5. Лог показал: **ни одно** угаданное имя не совпало с реальностью — в `services[]` заказов Ozon лежит только логистика, никакой рекламы там нет вообще. Вся гипотеза предыдущего фикса была неверна с самого начала — никто не проверил её по живым данным перед тем как мерджить.
6. Переписал `get_fin_adv_spend` в чисто диагностический режим (логирует operation_type верхнего уровня + опрашивает доп. transaction_type на случай если "all" не включает все типы), убрал бесполезный `_MARKETING_SERVICES`. Также снял фильтр `state=CAMPAIGN_STATE_RUNNING` в Performance API — короткие/завершённые CPC/CPO кампании раньше терялись.

## 3. Решили — частично

- Задвоение логистики в ДРР — решено (PR #9, смерджен).
- Корневая причина расхождения ДРР (Premium/бренд/CPO не покрыты API) — **не решена**, только улучшена диагностика. Нужен ещё один цикл: реальный синк → сырой лог → найти точные operation_type → зашить точные суммы.

## 4. Что можно было лучше

- **Главная находка**: фича `get_fin_adv_spend` с угаданными именами сервисов была смерджена в `main` без единой проверки на реальных данных Ozon API. Это ровно та же ошибка, которую правил исходный алерт (угадывать вместо проверки) — просто никто не поймал её до прода. Правило на будущее: для любой интеграции с внешним API, где совпадение полей по имени критично (а не просто типизация) — нельзя мерджить по одной лишь логике/документации, нужен хотя бы один прогон на реальных данных ДО мерджа, либо фича должна сразу разворачиваться в диагностическом режиме (логировать факт, не использовать результат), и только после подтверждения реальных значений — включаться в расчёт.
- Я сам в этой сессии сначала тоже почти повторил тот же паттерн (создать `_NON_MARKETING_SERVICES` и сразу смержить), хотя в этом случае был относительно низкий риск (только подавление алерта, не расчёт), это нормально.
- WebSearch/WebFetch не дали точных значений `operation_type` Ozon API (документация Ozon блокирует автоматический фетч, 403) — для проприетарных API с непубличной/блокируемой документацией единственный надёжный путь — эмпирика по живым логам, тратить время на поиск в интернете не стоило so much (потратил ~4 запроса WebSearch без результата).

## 5. Что узнал нового о проекте

- В Ozon API расходы на рекламу физически разделены на несвязанные источники:
  - CPC/CPO (клики, оплата за заказ) — Performance API кампании (`/api/client/campaign` + `/api/client/statistics`), НЕ финансовые транзакции.
  - Premium-подписка и продвижение бренда — неизвестно где (не нашли ни в `services[]`, ни (предположительно) через "all" агрегацию транзакций). Возможно эти 2 категории Ozon вообще не отдаёт через Seller API.
- `marketplace_financial_report.logistics/commission` уже корректно покрывают логистику/комиссии для NET-маржи — это отдельный, более надёжный пайплайн (`get_financial_report`, transaction_type=orders/returns), не путать с рекламным.
- Synced jobs: `_scheduled_adv_sync_loop` (03:00 UTC, дёргает `sync_ad_stats`) — ежедневно; `_scheduled_fin_sync_loop` (вс 01:30 UTC) — еженедельно, для NET-маржи, не для рекламы.
