# Рефлексия: NET-маржа WB и Ozon считалась по неполным/неверным данным

**Дата:** 2026-06-16
**Сессия:** запрос "покажи маржу КБ50 по МП за месяц" → разбор расхождения цены → три бага в пайплайне финотчётов

---

## 1. Какая задача была

Пользователь попросил показать NET-маржу товара КБ50 за месяц. Дальше — "распиши выплату по значениям на 1 товар" и финально "моя цена со скидкой — 541,44 р" (для WB) и "797 р" (для Ozon), что сразу выявило расхождение с расчётной ценой почти в 5 раз.

## 2. Как решали

1. **WB**: нашёл, что `WBClient.get_financial_report()` суммировал `quantity`/`revenue` по всем строкам отчёта `reportDetailByPeriod`, включая `doc_type_name == ""` (логистика, возмещение складских издержек) — у этих строк `quantity` ненулевой, а выручка нулевая, что разбавляло среднюю цену в ~5 раз. Подтвердил гипотезу прямым вызовом WB API и разбивкой по `supplier_oper_name` — точное совпадение раздутия (170 реальных продаж + 806 "паразитных" = 976, как в БД). Починил фильтром `doc_type_name == "Продажа"`.
2. **Ozon, баг 1**: синк не работал вообще — Ozon API не принимает период дольше месяца, а код слал 90 дней одним запросом; ошибка только логировалась, не выбрасывалась → `/sync_fin` молча давал 0 строк. Починил чанкингом по 28 дней.
3. **Ozon, баг 2**: после чанкинга всё равно 0 строк — код искал `offer_id` в `items[]`, а `/v3/finance/transaction/list` отдаёт там только `sku`. Переключил на `sku`, поправил join в `peter.py` (`m.ozon_sku` вместо `m.ozon_offer_id`), задокументировал в db-schema skill.
4. **Ozon, баг 3**: данные пошли, но `quantity`/`revenue` стабильно 0 — у этого эндпоинта в принципе нет полей `quantity` на уровне операции и `price`/`quantity` в `items[]`. Эмпирически (после провала прошлой сессии с угадыванием полей Ozon API — см. retrospective про ДРР) проверил `/v2/finance/realization`: там есть `seller_price_per_instance` и `delivery_commission.quantity` — причём `quantity` внутри строки может быть >1, это не "одна строка = одна штука". Добавил `OzonClient.get_realization_quantity_revenue()`, смерджил с payout/commission/logistics из transaction list по тому же `sku`.
5. Дату агрегата для realization (помесячный отчёт) сначала поставил на 1-е число месяца — оказалось, что стандартный фильтр "последние 30 дней" эти строки не захватывает (1 мая раньше окна 17 мая — 16 июня). Поправил на последний день месяца.
6. Удаление 3 устаревших строк с неправильной датой потребовало явного подтверждения пользователя — auto-mode classifier заблокировал DELETE по таблице финотчёта без него, и правильно: разрешение запросил, получил "да", выполнил.

## 3. Решили — да

NET-маржа КБ50 по обеим площадкам теперь соответствует реальной цене: WB ~533 ₽/шт (было ~110), Ozon 771 ₽/шт (было ~0, после первого фикса было бы ~0 из-за отсутствия quantity). Закоммичено (`78d19a7`), деплой оставлен на решение пользователя.

## 4. Что можно было лучше

- Сразу проверять реальный формат ответа внешнего API перед тем как чинить агрегацию "по логике" — для Ozon-бага 2 и 3 это сделал (живые вызовы API), для WB-фикса тоже подтвердил гипотезу эмпирически до правки кода. Это сработало хорошо — повторений паттерна "угадал по документации, не проверил" не было.
- Three bugs in one пайплайне нашлись только потому, что пользователь дал конкретную цифру (541,44 ₽, 797 ₽) для сравнения. Без эталонного числа от пользователя расхождение в 5х в среднем по списку товаров легко не заметить — агрегаты "выглядят правдоподобно" сами по себе. Стоит закладывать сверку с известной ценой как обязательный шаг validación при любых правках финансовых пайплайнов.
- Локальная среда без зависимостей проекта (`loguru`, `aiohttp`, `asyncpg` пришлось доустанавливать, `tools/__init__.py` тянет лишние пакеты типа `tavily`) — каждый раз обходить импорт пакета через `importlib.util` отдельным модулем. Стоит запомнить этот путь обхода для будущих диагностик без полного venv.
- Кодировка консоли Windows (cp866 vs UTF-8) ломала кириллицу в stdout — писать результаты сразу в UTF-8-файл и читать через Read оказалось надёжнее, чем пытаться поправить кодировку консоли.

## 5. Что узнал нового о проекте

- В `marketplace_financial_report` для Ozon исторически ожидался `offer_id` (как у WB), но реальный источник (`/v3/finance/transaction/list`) физически не может его дать — там только `sku`. Это тот же класс рассинхрона Ozon `offer_id`/`sku`, который уже чинили дважды в других местах (см. `retrospectives/2026-06-16_dashboard-sync-roas-ozon-id-mismatch.md`) — теперь чинили в третий раз, в финотчёте.
- `/v2/finance/realization` — единственный Ozon-эндпоинт с ценой за штуку и количеством; раньше в проекте не использовался вообще.
- WB-отчёт `reportDetailByPeriod` содержит вперемешку строки продаж и сопутствующих операций (логистика, компенсации) с одним и тем же полем `quantity`, различить их можно только по `doc_type_name`.

## 6. Нарушил ли агент правило молча?

Нет: DELETE по продакшен-таблице запросил подтверждение явно (через AskUserQuestion после блокировки auto-mode classifier), деплой не делал без команды. Один раз забыл удалить временный файл `out6.txt` из рабочей директории — заметил и почистил при следующей проверке `git status`.
