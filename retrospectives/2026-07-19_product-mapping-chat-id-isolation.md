# Изоляция product_mapping по chat_id — ретроспектива

**Дата:** 2026-07-19

## Какая задача была

Реализовать план `plans/2026-07-18-product-mapping-chat-id-isolation.md` целиком: сделать
схему `product_mapping` воспроизводимой из кода (Фаза 2) и включить изоляцию данных по
`chat_id` (Фаза 4), по запросу пользователя — без промежуточных подтверждений.

## Как решали

1. Фаза 1 (сверка схемы) — через `railway run --service Postgres python` (`psql` локально не
   установлен, `asyncpg` дал те же ответы). Нашли, что `chat_id` в `product_mapping`
   физически не существовал, вопреки `.claude/skills/db-schema/SKILL.md` — три функции в
   `db.py` уже обращались к несуществующей колонке и падали в проде (одна тихо глотала
   ошибку, другая ломала `/apply_prices` пользователю). Также подтвердили: бот
   single-tenant (один `chat_id` везде), backfill безопасен.
2. Реализацию (Фазы 2 и 4) делегировали параллельным субагентам по файлам: сначала `db.py`
   (схема + сигнатуры функций), затем одновременно `main.py`, `agents/max.py`,
   `agents/peter.py`, `agents/elina.py` — с точным списком новых сигнатур из первого шага,
   чтобы не дублировать исследование.
3. Каждый diff проверялся оркестратором перед тем, как считать фазу готовой — это поймало
   две проблемы, которые субагенты не должны были заметить по своим узким заданиям:
   неточный комментарий (утверждал, что `chat_id` "уже существовал" — не существовал) и
   сломанные `ON CONFLICT (display_name)` в трёх местах (`main.py`, `agents/max.py` ×2) —
   после смены уникальности на `(chat_id, display_name)` эти INSERT упали бы в проде.
4. Фаза 5 — без Docker/локального Postgres в этом окружении, прогнали `init_db()` на той же
   прод-БД, но в изолированной пустой Postgres-схеме (`search_path`, удалена после
   проверки) — тот же движок, без риска для реальных данных. Проверили создание схемы с нуля,
   владение `mapping_id` (IDOR), уникальность в рамках чата.

## Решили — да/нет/частично

Да, план реализован полностью, все фазы `[x]`, Статус `завершён`. Два найденных вне рамок
плана бага (`cmd_sync_sku` без chat_id, `_save_product`'s `ON CONFLICT (mapping_id)` без
`marketplace`) не трогали — заведён GitHub issue.

## Что можно было лучше

- Агент `implementer` из `.claude/agents/implementer.md` не был доступен как зарегистированный
  subagent_type в этой сессии — пришлось использовать `general-purpose` и вручную передавать
  ему текст правил из `implementer.md` в промпте. Стоит проверить, почему кастомные project
  agents не подхватываются автоматически.
- Стоило сразу проверить `ON CONFLICT (display_name)` по всему репо (`grep`) сразу после
  первого субагента (db.py), а не после того, как все 4 параллельных субагента уже
  отработали — нашли бы раньше и не пришлось бы держать это в уме параллельно с ревью
  четырёх diff'ов.

## Что узнал нового о проекте/пользователе

- Пользователь ожидает, что при "делай без подтверждений" оркестратор продолжает *сам*
  проверять каждый diff перед коммитом (это уже было в CLAUDE.md) — "без подтверждений"
  относится к тому, что не нужно спрашивать пользователя на каждом шаге, а не к тому, что
  можно пропускать собственную проверку качества.
- `.claude/skills/db-schema/SKILL.md` может расходиться с реальной прод-схемой (см. находку
  Фазы 1) — при работе со схемой БД стоит перепроверять через реальный `railway run`, а не
  доверять описанию в skill не глядя.
