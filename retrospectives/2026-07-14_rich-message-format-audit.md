# Аудит формата сообщений по всем агентам

## Задача

После фикса нечитаемых HTML-отчётов Макса (см.
`retrospectives/2026-07-14_html-report-readability-fix.md`) Борис попросил
найти вообще все места в проекте, где формат Rich Message может ломаться,
чтобы выдача агентов всегда была визуально корректной.

## Как решали

Запустили Explore-агента на разведку по всем 9 агентам и `base_agent.py` —
искали каждое место отправки текста в Telegram и сверяли: формат реального
контента (HTML-теги / GFM Rich Markdown / хардкод-таблицы с `\n`) против
способа отправки (`parse_mode`, через `_send_rich`/GFM или напрямую).

Главная находка разведки: прошлый фикс (`looks_like_html()` в `_notify_user()`)
защитил только путь «очередь задач → worker loop», но не путь «живой диалог»
(`BaseAgent.handle_message()`) — там Тина (системный промпт требует HTML на
все ответы) и Макс (`_HELP_TEXT` с `<b>` встроен в системный промпт) ломались
в свободном чате точно так же, как раньше ломался каталог товаров.

Составили план (`plans/2026-07-14-rich-message-audit-fixes.md`) из 5 фаз по
убыванию серьёзности, каждую фазу делегировали отдельному вызову
implementer-субагента, каждый diff проверяли сами перед коммитом:

1. **Корень**: `handle_message()` в `base_agent.py` — вынесли развилку
   HTML/Rich Markdown в общий `_send_agent_text()`, используют и
   `_notify_user`, и `handle_message`.
2. **Марта**: три «теневых» `reply()` (голос, фото, действия с картинками) и
   fallback без очереди слали текст в обход `_send_rich` вообще без
   `parse_mode` — пользователь видел сырые звёздочки. Переведены на
   `_send_agent_text`.
3. **Кевин**: результат после подтверждения PR уходил без `parse_mode`.
   Заодно точечный фикс у Питера — одиночная `*звёздочка*` (курсив) вместо
   задуманного жирного.
4. **Элина**: название/описание товара из ответа Клода вставлялись в
   HTML-превью без `html.escape()` — символы `<`/`>`/`&` могли сломать
   разметку. Добавили экранирование в точке вставки.
5. **Ева**: `_split_digest` искала точки разреза по markdown-заголовкам
   (`##`), которых в HTML-дайджесте не бывает — жёсткий разрез мог порвать
   `<blockquote expandable>` пополам. Переписали на HTML-осведомлённый
   разрез по границам `\n\n` с проверкой баланса тегов; сам проверил логику
   отдельным прогоном на синтетических примерах (два блока по 195 симв. —
   режется ровно по границе, тег не разорван; fallback с `logger.warning`
   для редкого случая одного блока длиннее лимита).

## Решили

Да, план закрыт полностью (все 5 фаз `[x]`, статус «завершён»). Сознательно
не стали трогать (см. раздел плана «Не берём в эту итерацию»): три разных
формата в разных хардкод-командах Питера (самосогласованы, не баг, а
стилистический долг) и потенциально мёртвый код в `utils/tg_format.py`
(`MARKDOWN_FORMAT_RULES`/`escape_mdv2`/`strip_mdv2`) — решили не удалять
без отдельной проверки использования, чтобы не рисковать в рамках этой задачи.

## Что можно было лучше

- Существующий тестовый набор (`tests/`) не удалось прогнать в этой песочнице
  из-за конфликта версий `cryptography`/`cffi` в системном Python — полагались
  на `py_compile` и точечные ручные симуляции логики (извлекая функции в
  отдельные скрипты без тяжёлых импортов). Стоит перепроверить `pytest` в
  Railway/CI перед мержем.
- В `agents/eva.py` вообще нет юнит-тестов — стоит когда-нибудь добавить
  `tests/test_eva_split_digest.py` на новую HTML-осведомлённую логику разреза,
  чтобы не полагаться только на ручную проверку.

## Что узнал нового о проекте

Одна и та же категория бага («формат контента не совпадает со способом
отправки», уже описанная в `ai-clone/feedback/format-prompt-matches-parse-mode.md`)
системно повторяется в проекте на разных уровнях: не только в промптах и
`parse_mode` конкретных команд, но и в «общих» путях доставки
(`_notify_user`, `handle_message`) и в собственных эвристиках форматирования
(`_split_digest`), которые были написаны под старый формат (markdown) и не
обновились при миграции конкретных агентов на HTML. При будущих миграциях
формата стоит явно грепать не только `parse_mode=`/`reply_text(`, но и любые
самописные функции разреза/экранирования текста — они тоже кодируют
предположения о формате.
