# AI Office — навигация по проекту

Карта проекта. Читай в начале каждой сессии.
**AI Office** — команда из 9 агентов на Claude API, управляемых через Telegram, для автоматизации маркетплейс-бизнеса (WB + Ozon).
Один Python процесс на Railway. Репо: `titboan/ai-office` (приватный). ОС разработчика: Windows / PowerShell.

---

## Архитектура

```
Telegram → Марта (router + chain planner)
├── одиночная задача → enqueue_task() → агент
└── цепочка → Claude API → JSON план → [подтверждение] → выполнение
```

## Принципы (нарушать только с явным обоснованием)

1. **Один Python процесс** — никаких микросервисов
2. **Postgres — источник истины** — задачи, состояния, результаты только там
3. **Notion — UI**, не transactional backend
4. **Агенты изолированы** — только через очередь, никаких прямых вызовов
5. **Человек подтверждает** (в сессии Claude Code): `railway up`, merge PR, любые рискованные необратимые действия (force push, удаление), разовые write-запросы к WB/Ozon API (цены, рекламные кампании и т.п.). Автоматические фоновые задачи Макса (отзывы/вопросы, синк рекламы, сток-алерты) — не требуют
6. **Простота важнее фреймворков** — никаких LangGraph, CrewAI, Celery, DSPy
7. **Эволюция, не переписывание** — если нужно изменить >40% файла или сломать публичный интерфейс, сначала спроси

## Правила репо

1. `git add` поимённо — никогда `git add .` или `git add -A` (оба в deny в `.claude/settings.json`)
2. **Ветка и PR**: если сессия/харнесс сама указала рабочую ветку (Claude Code на вебе/GitHub) — работай в ней и открывай PR в `main`, не пытайся пушить в `main` в обход. Если работаешь локально в CLI без указанной ветки — пушь прямо в `main` (Railway деплоит только с `main`)
3. Перед `git push` проверить `git log origin/<branch>..<branch> --oneline`
4. `.env` не читать целиком — `Select-String "^VAR=" .env` для одной переменной
5. Не хардкодить динамические данные (цены, лимиты API, расписания) — в `config.py` в секцию `CONSTANTS` в конце файла
6. Если нужная папка не существует (`retrospectives/` и др.) — создай и продолжай

## Дашборд (React Mini App, `dashboard/`)

Дашборд — зеркало данных Telegram-агентов. **При любом изменении полей/расчётов в агентах — обнови и дашборд.**

- Новое поле в `_collect_data_for_dashboard()` → добавь в `api.ts` и компонент в `charts/`
- Изменился расчёт (маржа/срочность/метрика) → обнови легенду/тултипы/цвет в компоненте
- Переименование поля → синхронно переименуй в `api.ts` и всех компонентах

Карта компонентов: `NetMarginTable.tsx` ↔ `net_margin` · `StockTable.tsx` ↔ `stock_velocity` · `RevenueChart.tsx` ↔ `revenue_by_day` · `AbcTable.tsx` ↔ `abc_data` · `MarginChart.tsx` ↔ `net_margin` (основной), `margin_wb`/`margin_ozon` (GROSS-фоллбэк, если `net_margin` пуст) · `OrdersChart.tsx` ↔ `orders_by_day`

## Большая фича = план в `plans/`

Создай план если: затронуто 2+ файла, нужен новый эндпоинт / агент / таблица, или ты сам не уверен в объёме. Сомневаешься — создай план.

Формат: имя `YYYY-MM-DD-название-задачи.md`, фазы с чек-боксами `[ ]`/`[x]`, строка `Статус: в работе / завершён`.

Не успел закончить фазу за сессию — закоммить то, что прошло проверку, отметь фазу `[~] в процессе` в файле плана, не оставляй незакоммиченный код.

## Оркестрация: реализацию делегируй субагентам Claude Code

Ты — оркестратор. Делегируй содержательную правку **кода продукта** (`agents/`, `tools/`, `utils/`, `dashboard/src/`, `main.py`, `config.py`, SQL-миграции) субагенту через Agent (`.claude/agents/implementer.md`), каждую фазу плана — отдельным вызовом. Сам делай только мелкую механическую правку (опечатка, константа, однострочный фикс) и работу с `plans/*.md`, `retrospectives/*.md`, самим CLAUDE.md.

Промпт субагенту — самодостаточный (конкретное место, что и зачем менять, ссылка на фазу плана); не дублируй в нём то, что уже в `implementer.md`.

После каждой фазы — сам: прочитай diff, прежде чем коммитить или переходить дальше. `git add`/`commit`/`push` делает оркестратор, не субагент. Diff разошёлся с планом — не коммить молча, поправь сам или перезапусти субагента с уточнением.

Спроси пользователя вместо того чтобы действовать: при настоящей неоднозначности требований, которую план не разрешает (спроси до старта реализации, в идеале ещё на этапе плана). Рискованные необратимые действия — см. Принципы, п.5. Не спрашивай про рутинные детали реализации (наименования, форматирование, равнозначные мелкие варианты) — это решает оркестратор/субагент сам.

## Цикл в конце каждой фичи

1. Найди ошибки → докажи примером → исправь
2. Что упустили: граничные случаи, валидация входных данных, обработка ошибок
3. Реальная проблема нашего проекта? Сверь с `plans/`
4. Свежий взгляд — перечитай diff как будто видишь впервые
5. Security review: `.env`, API-ключи, персональные данные пользователей
6. Менял формат/вывод в Telegram? Грепни все `parse_mode=` и `reply_text(` — формат в промпте агента обязан совпадать с `parse_mode` на всех путях отправки, иначе сырые `*звёздочки*`. Подробно: `ai-clone/feedback/format-prompt-matches-parse-mode.md`

Фича готова к пушу когда: цикл пройден, план обновлён `[x]`, diff проверен (PR создан, если работали в отдельной ветке — см. Правила репо, п.2).
Если остался известный edge case — создай issue в GitHub, не держи PR.
