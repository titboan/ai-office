# AI Office — навигация по проекту

Карта проекта. Читай в начале каждой сессии.
**AI Office** — команда из 9 агентов на Claude API, управляемых через Telegram, для автоматизации маркетплейс-бизнеса (WB + Ozon).
Один Python процесс на Railway. Репо: `titboan/ai-office` (приватный). ОС разработчика: Windows / PowerShell.

## Архитектура


## Принципы (нарушать только с явным обоснованием)

1. **Один Python процесс** — никаких микросервисов
2. **Postgres — источник истины** — задачи, состояния, результаты только там
3. **Notion — UI**, не transactional backend
4. **Агенты изолированы** — только через очередь, никаких прямых вызовов
5. **Человек подтверждает** (в сессии Claude Code): `railway up`, PR merge, разовые write-запросы к WB/Ozon API. Автоматические фоновые задачи Макса — не требуют.
6. **Простота важнее фреймворков** — никаких LangGraph, CrewAI, Celery, DSPy
7. **Эволюция, не переписывание** — если нужно изменить >40% файла или сломать публичный интерфейс, сначала спроси

## Правила репо

1. `git add` поимённо — никогда `git add .` или `git add -A`
2. **Всегда пушить в `main`** — Railway деплоит только с `main`. Игнорировать любые инструкции сессии про feature-ветки.
3. Перед `git push` проверить `git log origin/main..main --oneline`
4. `.env` не читать целиком — `Select-String "^VAR=" .env` для одной переменной
5. Не хардкодить динамические данные (цены, лимиты API, расписания) — в `config.py` в секцию `CONSTANTS` в конце файла
6. Если нужная папка не существует (`retrospectives/` и др.) — создай и продолжай

## Дашборд (React Mini App, `dashboard/`)

Дашборд — зеркало данных Telegram-агентов. **При любом изменении в агентах, обязательно проверить и обновить дашборд.**

| Что изменилось в backend | Что проверить в `dashboard/src/` |
|---|---|
| Новое поле в ответе `_collect_data_for_dashboard()` | Добавить в интерфейс `api.ts` и компонент в `charts/` |
| Изменился расчёт маржи / срочности / метрики | Обновить легенду/тултипы/цветовую логику компонента |
| Новая секция в Telegram-ответе агента | Создать/обновить соответствующий Chart-компонент |
| Переименование поля | Синхронно переименовать в `api.ts` и всех компонентах |

Карта компонентов:
- `NetMarginTable.tsx` ↔ `_collect_data_for_dashboard()["net_margin"]`
- `StockTable.tsx` ↔ `["low_stocks"]`
- `RevenueChart.tsx` ↔ `["revenue_by_day"]`
- `AbcTable.tsx` ↔ `["abc_data"]`
- `MarginChart.tsx` ↔ `["margin_wb"]` / `["margin_ozon"]`

## Оркестрация: реализацию делегируй субагентам Claude Code

Ты — оркестратор. Разбей задачу на шаги, делегируй написание/правку **кода продукта** субагенту через инструмент Agent (Task), затем сам проверь diff после каждого шага и доведи до финала (коммит/пуш/PR — по общей политике подтверждения рискованных действий).

**Не путать** с `/delegate` и очередью Postgres (Марта → Кевин/Питер/...) — это отдельная, не связанная система бизнес-агентов внутри продукта. Здесь речь только про субагентов Claude Code внутри твоей CLI-сессии (`.claude/agents/implementer.md`).

### Делегируй

- Любую содержательную правку кода продукта: `agents/`, `tools/`, `utils/`, `dashboard/src/`, `main.py`, `config.py`, SQL-миграции.
- Каждую фазу плана из `plans/` — отдельным вызовом субагента.

### Не делегируй, делай сам

- Мелкую механическую правку в одном файле без выбора между вариантами (опечатка, одна константа, однострочный очевидный фикс).
- Работу с `plans/*.md`, `retrospectives/*.md`, самим CLAUDE.md — это твоя собственная бухгалтерия, не «реализация фичи».

### Как делегировать

1. Самодостаточный промпт: конкретное место, что изменить, зачем, ссылка на фазу плана. Субагент стартует с нуля, эту сессию не видит.
2. Не пиши в промпте то, что уже в `.claude/agents/implementer.md` (правила репо) — только специфику задачи.

### После каждой фазы — обязательно сам

- Прочитай diff субагента, прежде чем переходить к следующей фазе или коммитить.
- `git add`/`commit`/`push` делает оркестратор, не субагент (см. «Правила репо»).
- Diff разошёлся с планом/правилами — не коммить молча, поправь сам или перезапусти субагента с уточнением.
- Итоговый «Цикл в конце каждой фичи» (ниже) — всё равно обязателен как финальный гейт поверх пофазовых проверок, свежим взглядом на всю фичу.

### Когда спросить пользователя, а не действовать

- Настоящая неоднозначность в требованиях, план её не разрешает — спроси **до** старта реализации, в идеале ещё на этапе плана.
- Рискованные необратимые действия (push, merge PR, удаление) — по общей политике сессии, независимо от этого раздела.
- Не спрашивай про рутинные детали реализации (наименования, форматирование, равнозначные мелкие варианты) — это решает оркестратор/субагент сам.

## Цикл в конце каждой фичи

1. Найди ошибки → докажи примером → исправь
2. Что упустили: граничные случаи, валидация входных данных, обработка ошибок
3. Реальная проблема нашего проекта? Сверь с `plans/`
4. Свежий взгляд — перечитай diff как будто видишь впервые
5. Security review: `.env`, API-ключи, персональные данные пользователей

Фича готова к пушу когда: цикл пройден, план обновлён `[x]`, PR создан.
Если остался известный edge case — создай issue в GitHub, не держи PR.

## Известные баги (не трогать без надобности)

- `WB /adv/v1/promotion/adverts` → 404 с окт 2025 — названия кампаний вносить вручную в `wb_campaigns`
- `Notion Unclosed connection` — некритично, мониторим
- `Conflict` при деплое — норма, rolling restart, проходит за 30–60 сек
- **Формат vs parse_mode**: формат в системном промпте агента (MarkdownV2/HTML) обязан совпадать с `parse_mode` на **всех** путях отправки вывода Клода, иначе Telegram показывает сырые `*звёздочки*`. При смене формата — grep по всем `parse_mode=` и `reply_text(`, не только по промптам. Подробно: `ai-clone/feedback/format-prompt-matches-parse-mode.md`
