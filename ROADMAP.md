## Принципы системы

1. **Один Python процесс**, пока это возможно — никаких микросервисов без реальной необходимости.
2. **Postgres — источник истины.** Все задачи, состояния, результаты — только там.
3. **Telegram — основной и единственный UI.** Notion удалён (июнь 2026): не использовался, лишняя зависимость.
4. **Каждый агент ничего не знает о других агентах напрямую.** Все взаимодействия через очередь задач.
5. **Человек подтверждает потенциально опасные действия** — деплой, PR, write-запросы WB/Ozon API.
6. **Простота важнее модных AI-фреймворков.** Никаких LangGraph, CrewAI, Celery, DSPy.
7. **Эволюция, не переписывание.** Каждое изменение — минимальный фикс поверх работающего.
8. **Telegram — основной канал управления.** Реальные данные живут в Postgres и Notion.

## Обзор

ИИ-офис с командой из 10 агентов на базе Claude API, управляемый через Telegram. Каждый агент — отдельный Telegram-бот. Все работают в одном Python процессе на Railway.

**Репозиторий:** <https://github.com/titboan/ai-office>
**Деплой:** Railway (один сервис, все агенты в одном процессе)
**Статус:** 🟢 Работает в продакшне

## Архитектура

```
Ты (Telegram: текст / голос / фото+подпись)
      │
      ▼
Марта (Router + Chain Planner + Claude Vision + кнопки)
      │
      ├── Фото без подписи → кнопки выбора действия (Redis: pending_image)
      │
      ├── Одиночная задача → enqueue_task() → агент
      │
      └── Цепочка → _plan_chain() → Claude API → JSON план
                        │
                        ▼
                   [🚀 Запустить?] — подтверждение
                        │
                        ▼
              Notion страница проекта (создаётся сразу)
                        │
                        ▼
       Agent[0] (напр. Кевин: create_repo → generate_image → design_system)
                        │ completed → _advance_chain()
                        ▼
       Agent[1] (напр. Кевин: HTML → Pages)
                        │ completed → _advance_chain()
                        ▼
       Финал → резюме + кнопки [🌐 Открыть сайт]

Postgres Tasks Queue
  ├── queued → acknowledged → running → completed/failed/timeout
  ├── priority (0/10/20), correlation_id, retry_count, remind_at
  ├── chain_id, chain_index, chain_total, chain_plan (JSONB)
  └── SELECT FOR UPDATE SKIP LOCKED (без race conditions)

Worker Agents (asyncio, поллинг каждые 2 сек по agent_key)
      │
      ▼
Claude API → результат в tasks.result → уведомление пользователю
```

## Ключевые петли автоматизации (Макс)

```
Питер /report
  → рекомендованная цена сохраняется в product_mapping
  → Макс /apply_prices → кнопки [Применить WB] [Применить Ozon] [Всё] [Отмена]
  → WB/Ozon API update_prices

CTR < 1% (дайджест Питера)
  → Redis pending_infographic:{chat_id}
  → Марта ждёт фото
  → Макс загружает инфографику на WB/Ozon
  → product_mapping.infographic_updated_at = NOW()
  → через 14 дней Питер показывает CTR до/после в /report

ДРР выходит за пределы (авто 03:30 UTC или ручной /bid_adjust)
  → кнопки [✅ Снизить ставку] [❌ Пропустить]
  → WBClient.update_campaign_cpm()
```

## Что работает

### ✅ Phase 1–7 (выполнено)

- 10 агентов в одном asyncio event loop (PTB 22.7)
- Postgres task queue, worker loop, retry, таймауты
- Redis история диалогов с автосжатием (Haiku summary при ≥15 сообщений)
- Correlation ID сквозной через всю цепочку

- Приоритеты, `remind_at`, ntfy.sh push, Groq Whisper голос
- Claude Vision, цепочки с подтверждением
- Макс: полный цикл WB+Ozon (отзывы, заказы, остатки, реклама, инфографика)

### ⚪ Phase 10 — Мульти-магазин и SaaS-фундамент

> SaaS (Software as a Service) — монетизация проекта: клиент платит ежемесячно и управляет своим AI Office через Telegram, не зная как устроено изнутри.

**Что уже готово:** `chat_id` = tenant ID, все данные изолированы по `chat_id`, `shop_id` разделяет данные между магазинами. Архитектура правильная.

**Межагентная коммуникация — Марта как единственный UI (нужно до SaaS):**

> Обновление 2026-07-14: **завершено.** Коммит `d54556f` («Фаза 4: отключить Telegram polling у
> 6 остальных ботов») отключил polling у всех агентов кроме Марты — по решению владельца, жёстко
> и сразу. `run_all_async()` в `main.py` теперь поднимает `start_polling_async()` только для
> `MartaAgent`, остальные — `start_worker_only_async()` (только фоновый worker loop: очередь,
> синки, дайджесты, алерты; без `Application`/`initialize`/polling). Клиент физически не может
> написать другому боту напрямую — Telegram не доставит апдейт, некому его принять. Подробности и
> статус по фазам — `plans/2026-06-24-single-entry-marta.md`.

Раньше каждый агент был отдельным Telegram-ботом со своими командами — клиент был вынужден знать
о Максе, Питере, Элине и т.д. Теперь так: **Марта = единственная точка входа**, остальные агенты —
headless workers без публичного бота (мелкая незавершённость — см. «Осталось» ниже, не влияет на
клиента).

```
Клиент → Мартин бот (единственный)
    │
    ├── /sync, /report, /reviews → enqueue_task() → Макс/Питер (воркер)
    │       воркер шлёт результат через notify_chat_id → Мартин бот → клиенту
    │
    ├── свободный текст → Марта роутит → агент выполняет → ответ через Мартин бот
    │
    └── callback-кнопки (rp:, bid:, price_apply:, pmenu:, pset: и др.) — зеркала
            зарегистрированы на Мартином App, делегируют напрямую агенту
```

## Известные проблемы

### ⚠️ Conflict при деплое — это нормально

При rolling restart Railway появляется:
```
telegram.error.Conflict: terminated by other getUpdates request
```
Обработано через `_on_polling_error` → `os._exit(0)`. Само проходит за 30–60 сек.

### ⚠️ WB /adv/v1/promotion/adverts → 404

Эндпоинт недоступен с октября 2025. Названия рекламных кампаний вносить вручную в таблицу `wb_campaigns`.

### ⚠️ Дэн заморожен

Pollinations.ai генерирует 30–120 сек. Замена: Replicate/Flux (~$0.003/img).

### ⚠️ Ева ждёт TELETHON_SESSION

Агент запускается, но дайджест каналов не работает без сессии Telethon.

### ❄️ Кевин заморожен (2026-07-04)

Проверили историю проекта (ретроспективы, планы за месяц) — Кевин ни разу не использовался
для реальной задачи (создание лендинга/сайта, PR). Единственная его функция дублируется
Claude Code сессиями напрямую — с полным контекстом репо, тестами и ревью, а не слепым
tool_use циклом внутри Telegram. Убран из `AGENTS` в `main.py`, из планировщика цепочек и
delegation-registry Марты (`agents/marta.py`); проксирующие команды `/code`, `/plan` теперь
честно отвечают "заморожен" вместо тихой постановки задачи в мёртвую очередь.

Код (включая approval-gate на create_pr от 2026-07-04) не удалён — разморозить можно,
раскомментировав `"kevin": (KevinAgent, "kevin")` в `main.py` и вернув соответствующие блоки
в `agents/marta.py` (см. git diff коммита разморозки). Разморозить, если появится реальный
повторяющийся сценарий (не разовый).
