# Потеря и искажение данных при синке Макса (WB/Ozon)

**Дата:** 2026-07-18
**Статус:** в работе

## Контекст

Аудит проекта (сессия 2026-07-18) нашёл девять багов в синхронизации данных с WB/Ozon
(`agents/max.py`, `tools/marketplace.py`): критичные отзывы могут потеряться навсегда,
пагинация отзывов/вопросов ограничена 100 позициями, `last_checked_at` никогда не
обновляется, два магазина одного маркетплейса у одного продавца затирают друг другу KPI и
остатки «в пути», а часть сетевых ошибок WB/Ozon (429/5xx, обрезанная пагинация заказов,
битая кодировка CSV) тихо режет или искажает данные без единого предупреждения в логах.

## Фазы

### Фаза 1 — ретрай для «зависших» pending_approval отзывов [x]
- `agents/max.py:1238-1249` (`process_reviews`) — статус `pending_approval` пишется в БД
  до попытки уведомления; если `_notify_pending` (max.py:3141) не смог доставить сообщение,
  отзыв больше никогда не считается «новым» на следующих синках. По аналогии с
  `process_questions` (max.py:1330-1369, там есть ретрай зависших `pending`) — добавить
  такой же повторный показ отзывов, застрявших в `pending_approval` дольше разумного времени.

### Фаза 2 — пагинация отзывов и вопросов [x]
- `tools/marketplace.py:86` (WB `get_new_reviews`), `:1150` (WB `get_questions`),
  `:1307-1308` (Ozon `get_new_reviews`) — сейчас всегда один запрос (`skip=0`/`offset=0`).
  Добавить цикл пагинации до исчерпания данных (или до разумного предела с явным логом об
  усечении), чтобы backlog >100 позиций (после отпуска, сбоя бота на несколько дней) не
  терялся навсегда.

### Фаза 3 — использовать `last_checked_at` по назначению [x]
- В `db.py` есть только сброс `last_checked_at = NULL` (строка 1580), нигде не проставляется
  реальное значение после успешного синка. Из-за этого `since` в `process_reviews`
  (`agents/max.py:1187-1191`) всегда `now() - 7 дней` вместо «с последнего чек-ина» —
  добавить проставление `last_checked_at = NOW()` после успешного цикла синка магазина.

### Фаза 4 — `shop_id` в ключах KPI и остатков «в пути» [ ]
- `db.py:1839-1864` (`upsert_shop_kpi`, ключ конфликта `(snapshot_date, chat_id, marketplace)`)
  и `db.py:704-711` (`marketplace_in_transit`, `PRIMARY KEY (chat_id, marketplace, product_id)`)
  — оба без `shop_id`, из-за чего второй магазин Ozon/WB у одного продавца затирает данные
  первого. Аналогичная проблема для `marketplace_stocks`/`marketplace_financial_report`/
  `marketplace_fin_adv` уже была исправлена (миграция `db.py:641-699`, `shop_id` добавлен в
  UNIQUE) — применить тот же паттерн к этим двум таблицам. Также поправить
  `agents/max.py:2778-2811` (`sync_shop_kpi`) — результат сейчас складывается в
  `results["ozon"] = kpi` (словарь по маркетплейсу, не по магазину), нужно по `shop_id`.

### Фаза 5 — WB реклама: не терять батч целиком на 429 [ ]
- `tools/marketplace.py:738-761` (`get_ad_stats`) — при 429 код делает `sleep(60)` и `continue`
  на **внешний** цикл по батчам (до 50 кампаний), из-за чего текущий батч теряется вместо
  повторной попытки. Добавить внутренний retry именно для текущего батча перед переходом
  к следующему.

### Фаза 6 — не резать пагинацию Ozon заказов/продаж молча [ ]
- `tools/marketplace.py:1708-1710, 1801-1803` (`get_sales`/`get_orders`) — жёсткий обрыв на
  `offset >= 2000` без единой строки в логе. Либо снять ограничение (пройти все страницы),
  либо оставить как защиту от аномалий, но обязательно логировать факт усечения с
  количеством оставшихся данных, чтобы это было видно в `/logs`, а не только «итого: N».

### Фаза 7 — устойчивость к нестандартной кодировке/формату CSV Ozon [ ]
- `tools/marketplace.py:2866` (`_extract_csv_texts_from_zip`) — жёсткий `utf-8-sig`. Добавить
  fallback на `cp1251` при неудачном парсинге/пустых заголовках.
- `tools/marketplace.py:2895-2899` — несовпадение `№\s*(\d+)` или `len(lines) < 2` сейчас
  молча `continue`. Добавить `logger.warning` с именем файла, чтобы пропажу целой пачки
  статистики было видно в логах (по аналогии с уже существующей диагностикой в
  `get_financial_report`, строки 903-907).

### Фаза 8 — retry на 5xx, не только 429, в ручных WB-циклах [ ]
- `tools/marketplace.py:235-245, 469-480, 513-524, 654-664, 1221-1244`
  (`get_stocks`/`get_orders`/`get_orders_all`/`get_sales`/`get_returns_analytics`) — проверяют
  только `resp.status == 429`, хотя докстринг модуля (строки 4-5) обещает retry на 429/500.
  Общий `_request()` уже умеет ретраить `{429, 500, 502, 503}` — перевести эти методы на
  него вместо ручных циклов, чтобы временный 500 от WB не превращался в «заказов нет».

### Фаза 9 — не путать временный сбой с невалидным токеном [ ]
- `tools/marketplace.py:166-183, 1337-1358` (`check_connection`, WB и Ozon) — таймаут/500
  трактуются как «токен неверен». Различить сетевую ошибку/5xx (сообщить «маркетплейс
  временно недоступен, попробуйте позже») от 401/403 (реально «переподключите магазин»).
