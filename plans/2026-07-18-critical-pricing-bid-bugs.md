# Критичные баги применения цены и рекламных ставок

**Дата:** 2026-07-18
**Статус:** в работе

## Контекст

Аудит проекта (сессия 2026-07-18, без отдельной ретроспективы — результаты в чате) нашёл
пять багов на пути «применить цену/ставку», которые касаются реальных денег продавца:
WB/Ozon API может отклонить цену, а бот всё равно рапортует об успехе; нет проверки цены на
адекватность; лок репрайсера снимается раньше времени; направление изменения ставки может не
совпадать с тем, что подписано в UI; поиск товара по артикулу не гарантирует уникальность.

## Фазы

### Фаза 1 — честный статус применения цены [x]
- `tools/marketplace.py:2537-2566` (Ozon `update_prices`) — читать `result[].updated`/
  `result[].errors` из ответа, а не считать успехом сам факт HTTP 200. Если Ozon отклонил
  конкретный offer_id — вернуть это отдельно, не сливать в общий `success`.
- `tools/marketplace.py:581-600` (WB `update_prices`) — WB Price API асинхронный
  (`uploadID` — это принятие задачи в очередь, не факт применения). Либо реализовать
  поллинг реального статуса задачи через отдельный WB-эндпоинт, либо (если поллинг
  нецелесообразен) явно переименовать/переформулировать результат как «отправлено на
  обработку» и не писать новую цену в БД до подтверждения.
- `agents/max.py:5621` (`_apply_price`) — использовать уточнённый результат из клиента:
  при частичном/полном отклонении не обновлять `wb_price`/`ozon_price` в БД, показать
  пользователю причину отказа, а не общее «✅ Цена обновлена».

### Фаза 2 — sanity-check цены на обоих путях ввода [ ]
- Ручной ввод в Telegram (`agents/max.py:5606`, `raw.isdigit()`) — добавить верхнюю границу
  (например, не более чем в N раз выше текущей цены) и сравнение с себестоимостью
  (`product_costs`, если задана) — предупреждать/блокировать явно абсурдные значения
  (0₽, лишний ноль).
- Дашборд (`main.py:983`, `_handle_apply_price`) — та же проверка, не только `new_price <= 0`.
- Ограничения держать в одном месте (общая функция/константа в `agents/max.py`), чтобы
  Telegram и дашборд не могли разойтись — оба пути в итоге вызывают один и тот же
  `_apply_price`.

### Фаза 3 — валидация direction/delta_pct для ставок [ ]
- `main.py:1126-1148` (`_handle_apply_bid`) — проверять знак и разумный диапазон `delta_pct`
  (сейчас только `direction in ("up","down")`, само число не проверяется).
- `agents/max.py:6859-6900` (`_apply_wb_bid_raw`/`_apply_ozon_bid_raw`) и подпись в UI
  (`agents/max.py:6268`, «📉 Снижены» и т.п.) — формировать текст по фактическому знаку
  изменения ставки (до/после), а не по заявленному `direction`, чтобы при отрицательном
  `delta_pct` подпись не противоречила реальному результату.

### Фаза 4 — лок `/reprice` снимать только когда разобраны все карточки [ ]
- `agents/max.py:5479-5591` (`cmd_reprice`/`_handle_reprice_callback`) — сейчас
  `reprice_lock:{chat_id}` снимается безусловно на первом же «Применить» (строка 5591).
  Нужно снимать лок только когда обработаны все карточки батча (например, считать в Redis
  оставшиеся необработанные карточки, аналогично `count_incomplete_in_group` в
  `task_queue.py`) — иначе новый `/reprice` может запуститься поверх ещё не разобранного
  старого батча.

### Фаза 5 — устранить неоднозначность поиска `wb_nm_id` по артикулу [ ]
- `agents/max.py:5638` — `SELECT wb_nm_id FROM product_mapping WHERE wb_article = $1` без
  дедупликации. Проверить, может ли в реальных данных быть больше одной строки с одним
  `wb_article` (см. `plans/2026-07-18-peter-analytics-accuracy.md`, Фаза 5 — вопрос
  уникальности `product_mapping` шире одного этого места). Минимально: если `fetchrow`
  находит более одной строки — логировать предупреждение и не применять цену вслепую к
  случайной из них.
