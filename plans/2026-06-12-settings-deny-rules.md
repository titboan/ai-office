# Добавить deny-правила в settings.json

Статус: завершён

## Контекст

CLAUDE.md содержит критичные запреты (`git add .`, `railway up` без команды, чтение `.env` целиком),
но они существуют только как текст — агент может нарушить их случайно без предупреждения.
Цель: перенести самые опасные запреты в `settings.json` так, чтобы harness блокировал их
автоматически, не полагаясь на то, что агент прочитает и запомнит правило.

---

## Что создать

Новый файл: `C:\Users\Дом\ai-office\.claude\settings.json`
(проектный уровень — глобальный `C:\Users\Дом\.claude\settings.json` не трогать, там только `effortLevel`)

## Содержимое файла

```json
{
  "permissions": {
    "deny": [
      "Bash(git add -A*)",
      "Bash(railway up*)"
    ]
  }
}
```

---

## Почему именно эти два

| Правило | Почему deny, а не текст |
|---------|------------------------|
| `git add -A` | Необратимо стейджит всё, включая .env — высокий риск утечки |
| `railway up` | Деплой в прод без команды Бориса — явный запрет в Принципе 5 |

## Почему `git add .` НЕ включаем

Паттерн `Bash(git add .*)` совпадёт с `git add .gitignore` — легитимная команда.
Борис принял решение: оставить только текстовое правило в CLAUDE.md, не блокировать через settings.json.

## Что НЕ включать

- Чтение `.env` целиком (`cat .env`, `Get-Content .env`) — слишком агрессивно,
  легитимные случаи есть (проверка наличия файла, структуры). Достаточно правила в CLAUDE.md.
- `git push --force` — уже покрыт глобальными правилами Claude Code.
