# Живой диалог: прозрачность допущений и уточнения только там, где дорого ошибиться

Статус: в работе

## Контекст

Разбирали, насколько гибко агенты ведут диалог: задают ли уточняющие вопросы, когда задача
неясна, или молча гадают. Нашли:

- Уточнение (`clarification_needed`) есть только у Марты в `_plan_chain` — и только для
  write-операций без конкретики («измени цены», «удали карточки»). Для отчётов/аналитики
  уточнять явно запрещено промптом.
- `request_clarification()` в `base_agent.py` — метод для агентов, чтобы спросить пользователя
  посреди задачи. Нигде не вызывается — мёртвый код.
- Свободный текст без слэш-команды (например, «как дела с продажами») у Питера молча берёт
  дефолтный период (`agents/peter.py:981`, `_days = 14`, если в тексте нет «за N дней») и не
  говорит об этом пользователю в ответе.
- Проверили Макса: costly write-действия (создание рекламной кампании, применение новых цен)
  **уже** защищены inline-кнопками подтверждения (`/new_campaign`, `/reprice`,
  `cmd_apply_prices`), а свободный текст про кампанию явно отказывается гадать и отправляет
  на слэш-команду (`agents/max.py:397-398`). Здесь всё уже сделано правильно — не трогаем.
- У Алекса создание напоминания идёт через tool-loop (`_run_with_tools`,
  `_run_create_reminder_tool`): если Claude не может уверенно разобрать дату/время, он либо
  подставляет ISO-дату сам (может ошибиться на день/время), либо получает ошибку парсинга и
  перевыбирает — но никогда не спрашивает пользователя открытым текстом.
- Каспер отдаёт в поиск сырой текст задачи и не проговаривает, как понял расплывчатый запрос,
  прежде чем выдать отчёт.
- **Важное архитектурное ограничение:** единственная точка входа в Telegram — Марта
  (`start_worker_only_async` у остальных: без polling, только worker loop на очереди задач).
  Значит агент в фоне (Питер, Каспер, Алекс) не может просто «ответить вопросом» в диалоге — у
  него нет прямого канала к пользователю. Ответ пользователя всегда прилетает через
  `MartaAgent.handle_message`, и без явного сохранения через `_save_clarification`/
  `_pop_clarification` Марта не свяжет этот ответ с вопросом конкретного агента, а обработает
  как новый независимый запрос. Поэтому там, где агенту действительно нужно спросить (а не
  просто проговорить допущение в готовом ответе) — это обязана быть Redis-передача
  (`request_clarification()`/`_save_clarification()`), а не текстовый вопрос «в воздух».

**Осознанное решение — не делаем:** не строим многораундовую машину состояний для уточнений и
не расширяем `clarification_needed` на отчёты/аналитику. Асимметрия «не уточняем для read-only,
уточняем для write» — это разумная калибровка риска (ошибочный отчёт — просто попросишь другой,
ошибочная цена/кампания — реальные потери), её не трогаем. Проблема не в этом, а в том, что при
дефолте агенты **молчат** о допущении, и в паре мест (Алекс) **угадывают** там, где дешевле
спросить.

## Фазы

- [ ] **Фаза 1**: Питер — проговаривать дефолтный период/срез вслух
  - В `agents/peter.py::handle_task`, ветка свободного текста (~строки 1037-1073): когда
    `_days_match` не найден (период не указан явно пользователем), добавить в `prompt`
    инструкцию: «Период не указан явно — использован дефолт {_days} дней. Обязательно укажи
    это одной строкой в начале ответа и предложи уточнить, если нужен другой период».
  - То же для marketplace-среза, если в `_collect_data`/`_collect_advanced_data` есть неявный
    выбор «WB+Ozon» по умолчанию без явного запроса пользователя на оба — проверить и
    аналогично проговорить.
  - Не трогать slash-команды (`/report`, `/drr` и т.д.) — там период уже явный параметр с
    дефолтом, видимым в `/help`.

- [ ] **Фаза 2**: Каспер — проговаривать интерпретацию расплывчатого запроса
  - В `agents/kasper.py::handle_task` и/или `KASPER_SYSTEM`: если исходная задача очень
    короткая/расплывчатая (эвристика — например, короче ~40 символов без конкретики типа
    названия темы/периода/региона), добавить в промпт Claude инструкцию открыть ответ строкой
    «Понял запрос как: ...» перед структурированным отчётом. Не делать это для явно
    конкретных запросов — не превращать в занудную привычку.

- [ ] **Фаза 3**: Алекс — спрашивать через Redis-передачу вместо угадывания при нераспознанной
  дате напоминания
  - Алекс работает только в фоне (`start_worker_only_async`, нет своего Telegram polling) —
    значит просто вернуть текстовый вопрос из `_run_with_tools` недостаточно: этот текст уйдёт
    пользователю как обычный результат задачи, а его ответ придёт в `MartaAgent.handle_message`
    без связи с вопросом. Нужна явная передача через уже существующий механизм
    `_save_clarification`/`_pop_clarification` (`base_agent.py:347-394`), который Марта уже
    умеет подхватывать (`marta.py:786-807`).
  - Добавить в `ALEX_TOOLS` инструмент `ask_clarification(question: str)` рядом с
    `create_reminder`. В `ALEX_SYSTEM` — правило: если время/дата сформулированы так, что
    нельзя уверенно восстановить конкретный ISO-момент (в отличие от «через час», «завтра в 9
    утра» и т.п. однозначных формулировок) — вызвать `ask_clarification` вместо
    `create_reminder` с угаданным значением.
  - Обработчик вызова `ask_clarification` (в дополнение к `_run_create_reminder_tool`, там же
    где диспетчеризация `tool_name`): вызвать `await self._save_clarification(chat_id,
    question, original_payload=user_message)` (НЕ `request_clarification()` целиком — та ещё
    сама шлёт уведомление, а результат тула и так вернётся пользователю через обычный путь
    завершения задачи/ответа) и вернуть текст вопроса как финальный ответ `_run_with_tools`,
    завершив цикл (не продолжать crafting дальше).
  - Учесть оба пути вызова Алекса: прямое сообщение в чат (`handle_message` → напоминание
    создаётся сразу же, синхронно) и делегирование через очередь Марты (`handle_task`) — в
    обоих `chat_id` должен быть реальным ID пользователя на момент вызова `_save_clarification`.

## Критерий готовности

- Ручная проверка в Telegram (или через `handle_task`/`think` напрямую): расплывчатый запрос к
  Питеру без периода — в ответе явно назван взятый период. Расплывчатый запрос Касперу — в
  ответе явно назвал, как понял тему. Нечёткое напоминание Алексу («напомни как-нибудь на
  следующей неделе») — Алекс спрашивает, а не создаёт напоминание с угаданной датой.
- Ничего не меняем в существующей асимметрии уточнений Марты и в защищённых кнопками write-путях
  Макса — эти места уже устроены правильно.
