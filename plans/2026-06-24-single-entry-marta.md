# Марта как единственная точка входа

Статус: в работе (Фазы 1, 3, 4 закрыты фактически; Фаза 4 требует проверки на реальных
ботах после деплоя — см. чек-лист в конце документа; чек-лист общий актуализирован 2026-07-02)

Задача: пользователь устал переключаться между 8 ботами. Нет единой истории. Нужна единственная точка входа — чат Марты — где работают все команды всех агентов.

---

## Контекст и текущее состояние

Уже работает:
- `_ORIGIN_TOKENS["marta"] = MARTA_BOT_TOKEN` в `base_agent.py` — если задача поставлена Мартой (`from_agent="marta"`), результат идёт через бот Марты
- Марта уже роутит свободный текст к агентам через `enqueue_task`
- Уже есть прокси-команды в Марте: `/report`, `/sync`, `/reviews`, `/research`, `/write`, `/remind`

Чего не хватает:
- Прокси-команды Марты — не все (нет `/order`, `/supply`, `/drr`, `/abc`, `/funnel`, `/audit` и т.д.)
- Когда агент отправляет `InlineKeyboardMarkup`, `CallbackQueryHandler` зарегистрирован в Application агента, а не Марты → кнопки не работают через Марту
- Wizards Max (`/add_shop`, `/cost`, `/reprice`, `/map`) — ConversationHandler в Application Max

---

## Архитектура после рефакторинга

```
Пользователь → Мартин бот (единственный)
    │
    ├── /report, /sync, /reviews, /order, /supply... → enqueue_task(agent_key) → воркер агента
    │       воркер: _notify_user(chat_id, result, bot_token=MARTA_BOT_TOKEN)  ← уже работает
    │       callback кнопок: зарегистрированы на Мартином App                 ← нужно сделать
    │
    ├── свободный текст → Марта роутит → агент → результат через Марту       ← уже работает
    │
    └── Wizards (/add_shop, /cost, /reprice) → ConversationHandler на Марте   ← сложнее всего
```

## Фаза 2 — Callback-кнопки через Марту [~] частично

Проверено по коду 2026-07-02. Реализован **Вариант B по факту** (без общего `register_callbacks()`):
на Application Марты напрямую зарегистрированы `CallbackQueryHandler` с теми же паттернами,
что и у агента, а тонкая обёртка-метод Марты делегирует в `self._max_agent._handle_xxx(update, context)`
(пример: `marta.py:2047 _handle_camp_callback`). `context.bot` внутри — бот того Application,
которое реально диспетчит апдейт, так что кнопка на сообщении от Марты работает через Марту,
а на сообщении от Макса/Питера напрямую — через их бот. Патчить `self.bot_token` внутри
обработчиков не пришлось, т.к. они везде используют `query.message.reply_text` / `context.bot`.

### Перенесено (работает через Марту)
- [x] Max: `rev:` — approve/reject отзыв (`/reviews`, `/pending`)
- [x] Max: `qrev:` — вопросы
- [x] Max: `camp:`, `ozbid:`, `campnew:` — рекламные кампании и ставки Ozon
- [x] Max: `promo:` — акции
- [x] Max: `onboard:` — подключение магазина
- [x] Elina/Max: `seoapp:` — применение SEO
- [x] Питер/Марта: `msset:` — настройки поставок (заменили `pset:`, см. коммит 80bcfda)

### НЕ перенесено — но это не «сломанные кнопки», а UX-пробел
Ниже — кнопки, которые генерируются **только** внутри прямых `CommandHandler`
на собственном боте агента (`cmd_report`, `cmd_menu` и т.п.), а не в коде, который
исполняется при проксировании через `enqueue_task`. Поэтому у пользователя Марты
эти кнопки просто не появляются вовсе (не «висят» и не ловят несуществующий handler) —
но и быстрых next-step кнопок после `/report`/`/drr` он не получает, в отличие от
пользователя, который пишет напрямую `@peter_bot`/`@max_bot`.
- [ ] Peter: `pmenu:*` — кнопки `/menu` Питера (у Марты уже есть свой `/menu` → `mmenu:*`
      с тем же функционалом, так что дублировать необязательно — скорее отключить `/menu`
      у Питера, когда дойдём до Фазы 4)
- [ ] Peter: `pnext:*` — быстрые кнопки «Что дальше?» после `/report`, `/drr`
- [ ] Peter: `pseo:*` — кнопка запуска SEO у Элины после `/seo_audit`
- [ ] Peter: `returns_elina:*` — кнопка «улучшить карточку» после `/returns`
- [ ] Max: `price_apply:`, `rp:`, `costpick:`, `addmp:`/`addcat:`, `bid:`, `menu_` —
      частично устаревший `/start`-визард Макса, частично прайсинг/себестоимость

**Результат Фазы 2 (фактический):** критичные кнопки (отзывы, кампании, онбординг, настройки
поставок) работают через Марту. Оставшиеся — второстепенный UX (быстрые ссылки/меню),
не блокируют SaaS-запуск и не являются багом. Если решим их доделывать — переносить
тем же паттерном (тонкая обёртка в `marta.py` + `CallbackQueryHandler` с тем же паттерном),
понадобится добавить `marta_agent._peter_agent = peter_agent` в `main.py` (сейчас такой
связи нет, только `_max_agent`).

## Фаза 3 — Мастера Макса (wizards) [x] — решено иначе, чем планировалось

Изначальный план — переносить `ConversationHandler` из Max.app в Marta.app — не понадобился.
Вместо этого (см. `retrospectives/2026-06-26_ozon-full-management-saas-entry.md`):
- `cmd_proxy_*` в Марте дёргает `max_agent.cmd_xxx.__func__(max_agent, update, context)` —
  вызывает исходную функцию-визард напрямую в контексте Марты
- Состояние шага хранится в Redis (`onboard-сессия`), а не в стейт-машине PTB `ConversationHandler`
- Перехват последующего текстового ответа — в `handle_message` Марты, проверяет Redis-флаг

- [x] `/add_shop` → max, работает через Марту
- [x] `/cost` → max, работает через Марту
- [x] `/map` → max, работает через Марту
- [x] `/reprice` → max, работает через Марту
- [x] `/camp` → max, работает через Марту

**Известный пробел (из ретроспективы 2026-06-26):** `_handle_edit_reply` (режим редактирования
цен, MessageHandler group=1 у Макса) не перехватывается онбординг-сессией Марты — если
пользователь идёт через Марту и отвечает на сообщение с ценами, это не сработает.
Редкий сценарий, отложено.

**Результат Фазы 3:** все основные wizards работают через чат Марты.

## Фаза 4 — Убрать отдельные боты [ ] — сознательно не начата

Пересмотрено 2026-07-02: Борис пользуется Мартой чаще, чем остальными ботами напрямую —
можно закрывать личный доступ уже сейчас, без ожидания «первого внешнего клиента».
Реализовано **не буквально** по чек-листу ниже (полная остановка polling не понадобилась),
а более лёгким и обратимым способом — см. «Что сделали по факту».

### Что сделали по факту [x]

Вместо остановки `start_polling_async()` (что физически разрывает polling и потребовало бы
разносить его от `_worker_loop()`, которые сейчас стартуют вместе) — оставили ботов агентов
в polling, но перестали регистрировать у них личные Telegram-хендлеры:

- `BaseAgent.direct_commands_enabled: bool = True` (`agents/base_agent.py`) — новый флаг
- `build_app()`: если `False` → вместо `/start`, `/help`, `/reset`, MessageHandler(TEXT/VOICE)
  регистрируется один `_redirect_to_marta` (личка → «работай через Марту»); `_bot_commands()`
  не публикуется (`set_my_commands([])`)
- `_register_extra_handlers()` — теперь **всегда** вызывается (даже при `direct_commands_enabled=False`),
  чтобы агент мог зарегистрировать то, что не зависит от личного доступа (см. Макс)
- Flag выставлен `= False` у: Kevin, Kasper, Elina, Alex, Tina, Peter — их `_register_extra_handlers()`
  получили `if not self.direct_commands_enabled: return` в начале — 40+ личных команд не регистрируются
- Max — тоже `direct_commands_enabled = False`, но `_register_extra_handlers()` при выключенном флаге
  регистрирует ТОЛЬКО группу партнёров (`_handle_group_message` text+voice, `filters.ChatType.GROUPS`) —
  единственная причина, почему бот Макса вообще остаётся online (см. поднятый ранее блокер:
  реакция на «Макс»/упоминание в группе работает только пока бот физически принимает апдейты)
- Marta — без изменений (`direct_commands_enabled = True` по умолчанию)

**Результат:** личные команды у Kevin/Kasper/Elina/Alex/Tina/Peter/Max отключены, все они видят
«работай через Марту» на любое сообщение в личке. Бот Макса продолжает polling ради группы
партнёров. Worker loop (фоновые задачи из очереди) не тронут — работает у всех как раньше.

**Не сделано (сознательно, не требовалось для цели):**
- [ ] Физическая остановка polling / удаление `Application` у агентов — не нужно: боты просто
      ничего не делают с личными сообщениями, ресурсы на это минимальны, а разносить
      `start_polling_async()`/`_worker_loop()` — отдельная более рискованная переделка ради
      экономии, которая не была целью
- [ ] Убрать токены из Railway env (`PETER_BOT_TOKEN` и т.д.) — токены всё ещё нужны, боты
      всё ещё живы (polling), только не отвечают на личку. Трогать Railway env не потребовалось
- [ ] Обновить ROADMAP.md — сделать отдельным шагом после проверки на реальных ботах

**Проверить на реальных ботах перед тем, как считать законченным:**
- [ ] Написать `/start` любому из отключённых ботов (Кевин/Каспер/Элина/Алекс/Тина/Питер) —
      должен прийти редирект-текст, а не тишина и не старое поведение
- [ ] Написать личным сообщением боту Макса — тоже редирект
- [ ] Написать «Макс, ...» в группе партнёров — должна сработать реакция как раньше
- [ ] Убедиться, что фоновые задачи (дайджесты, синки) по расписанию продолжают приходить
      через бота Марты как обычно
