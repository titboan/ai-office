# Онбординг за руку: от токенов до готовой аналитики без единой команды наизусть

Статус: в работе

## Проблема

Сейчас после подключения токенов (`_send_finish`, `agents/max.py:690`) Макс просто
показывает текстовую памятку с командами (`/sync`, `/products`, `/dashboard`), а
`/sync_fin`, `/sync_adv`, `/sync_funnel` вообще не упоминает. Пользователь, который
не знает внутренней логики проекта, не понимает, что нужно вручную вызвать 4 разные
команды, чтобы в дашборде появились реальные заказы, ДРР, NET-маржа и воронка.
Плюс себестоимость (без неё маржа считается неверно) требует, чтобы товар был в
`product_mapping`, а туда он попадает только вручную через `/add` — то есть новый
пользователь должен сам додуматься вбить каждый товар руками, хотя Макс после
синка уже знает его товары и артикулы.

Цель: после ответа на вопросы мастера онбординга пользователь больше не гадает,
что делать — Макс сам подтягивает данные, сам заводит товары в реестр и сам,
товар за товаром, спрашивает себестоимость. Финал — сообщение с реальными цифрами
и рабочим дашбордом, без единой команды, которую нужно было ввести руками.

## Решение (согласовано с пользователем)

- Реестр товаров (`product_mapping`) заполняется автоматически из данных синка
  (остатки/заказы содержат `product_name` + `product_id` по каждой площадке) —
  без ручного `/add`.
- Кросс-площадочный мэтчинг WB↔Ozon делаем ТОЛЬКО по точному совпадению
  `display_name` (без фаззи-сравнения — примеры типа wb=`БК50гр` / ozon=`КБ50`
  показывают, что автоматически их сопоставить нельзя, а неверный мэтчинг склеит
  себестоимость разных товаров). Не совпало — заводим отдельными строками,
  пользователь сможет объединить позже вручную через `/map`.

## Фаза 1 — Единый прогон синка с видимым прогрессом

Файл: `agents/max.py`.

- [x] Новый метод `_run_full_sync_with_progress(chat_id, target_chat_id, bot)`:

- [x] После каждого шага — короткое сообщение с реальным результатом

      Ozon 8». Не одно "простыня"-сообщение в конце, а по шагу за раз, чтобы
      было видно, что процесс идёт.
- [x] Каждый шаг — в своём `try/except`: ошибка одной площадки/отчёта не должна
      прерывать всю цепочку. Явно показываем ⚠️ с причиной (см. принцип
      `_format_funnel_result` — не маскировать ошибку под успех).
- [x] Метод возвращает агрегированную сводку (кол-во заказов, товаров с

оркестратора после ревью субагента: убран `parse_mode="HTML"` из 4
error-веток, где в текст подставляется сырой `{e}` — иначе `&`/`<` в тексте
исключения (частые в URL от aiohttp) роняли бы саму отправку сообщения об
ошибке в Telegram.

## Фаза 2 — Автосоздание product_mapping из данных синка

Файлы: `db.py`, `agents/max.py`.

- [ ] Новая функция в `db.py`: `auto_populate_product_mapping(chat_id) -> dict`
      (возвращает `{"created": N, "merged": N}`).
      - Источник: `marketplace_stocks` (уже содержит `product_name` +
        `product_id` на площадку) для `chat_id`; если после `/sync` там пусто —
        fallback на `marketplace_orders`.
      - WB: `product_id` → `wb_article`. Ozon: `product_id` → `ozon_offer_id`.
      - Если в `product_mapping` уже есть строка с этим `wb_article`/
        `ozon_offer_id` — пропустить (уже заведено).
      - Если есть строка с ТОЧНО совпадающим `display_name` (без учёта
        регистра) и пустым нужным полем (`wb_article IS NULL` /
        `ozon_offer_id IS NULL`) — заполнить это поле (мэтч между площадками).
      - Иначе — `INSERT` новой строки, `display_name = product_name` (или
        `product_id`, если имени нет). При конфликте `UNIQUE(display_name)`
        (два разных товара с одинаковым текстом от площадки) — добавить
        короткий суффикс площадки к `display_name`, не терять товар молча.
- [x] Вызывается из `_run_full_sync_with_progress` сразу после шага
      "заказы/остатки" (Фаза 1, шаг 1) — товары должны быть в реестре до того,
      как дойдём до себестоимости.

Готово (`db.py:2263` `_auto_populate_side`, `db.py:2362`
`auto_populate_product_mapping`, обёртка `agents/max.py:2231`
`_auto_populate_products`). Пока НЕ вызывается из `_run_full_sync_with_progress`
— это сделает Фаза 4, чтобы не пересекаться правками с Фазой 1 в одном файле.

⚠️ Обнаружено при ревью: `product_mapping.display_name` UNIQUE — ГЛОБАЛЬНЫЙ,
не привязан к `chat_id`, хотя колонка `chat_id` в таблице есть (используется
только в `save_price_recommendations`/`get_price_recommendations`). Каталог
товаров сейчас общий на все чаты, которые подключены к боту — это существующее
поведение (`cmd_map`, `find_product_id_in_text`, `stocks_unified` VIEW тоже не
фильтруют по `chat_id`), не новая проблема этой фичи. `auto_populate_product_mapping`
намеренно повторяет эту же конвенцию (не пишет `chat_id` при INSERT), чтобы не
разъезжаться с остальным кодом. Если у бота когда-нибудь появится больше одного
независимого магазина от разных людей (а не один бизнес с несколькими chat_id) —
это станет реальным багом (чужие товары/себестоимость будут видны друг другу).
Не чиним в рамках этой фичи — вне её объёма.

## Фаза 4 — Хук после онбординга + финальное сообщение

Файл: `agents/max.py`.

- [x] `_send_finish` (max.py:692) вместо статичного текста-памятки:
      1. короткое «🔄 Собираю данные твоего магазина, 1-3 минуты…»
      2. вызов `_run_full_sync_with_progress` (Фаза 1, включает Фазу 2 —
         auto_populate вписан на место placeholder-комментария)
      3. если есть товары без себестоимости — сразу мастер себестоимости
         (Фаза 3); если реестр пуст — отдельное сообщение «в каталоге пока

      4. `_handle_cost_wizard_text` в конце мастера уже шлёт «Себестоимость
         задана для N из M» + кнопку дашборда — отдельного финального
         сообщения в `_send_finish` не нужно.
- [x] Убрать из текста финального сообщения перечисление команд
      (`/sync`, `/products`, `/dashboard` руками) — теперь это не нужно
      пользователю знать.
- [x] `/start` при повторном заходе (магазины уже есть) — без изменений,
      Фазы 1-3 запускаются только при первом подключении и при `add_shop`.
- [x] Подключён роутинг: `handle_message` (max.py:573) проверяет
      `_handle_cost_wizard_text` раньше состояния онбординга; `cmd_cancel`
      теперь чистит и `cost_wizard:{chat_id}`.

⚠️ Критичный фикс оркестратора после ревью: субагент изначально сделал
`_send_finish` синхронной — `await`-ила весь `_run_full_sync_with_progress`
(может занимать 1-3 мин из-за WB rate-limit sleep 60с) прямо в обработчике
Telegram-апдейта. PTB `Application` собран без `concurrent_updates=True`
(`base_agent.py:1144-1149`) → апдейты обрабатываются строго последовательно,
и на время синка бот вставал бы для ВСЕХ чатов (одобрение отзывов, другие
команды, ИИ-чат в группах). Вынес синк+мастер себестоимости в отдельный
метод `_run_post_onboarding_flow`, запускается через `asyncio.create_task`
(тот же паттерн, что уже используется в файле для ДРР через Питера,
max.py:6809) — обработчик апдейта возвращается сразу после приветствия,
остальное идёт в фоне.
