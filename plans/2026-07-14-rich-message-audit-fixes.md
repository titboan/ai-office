# План: привести формат сообщений в порядок у всех агентов

Статус: завершён

## Контекст

После фикса `_notify_user()` (см. `plans/2026-07-14-html-report-delivery-fix.md`)
Борис попросил найти ВСЕ места в проекте с той же категорией бага — «формат
контента не совпадает со способом отправки» (правило уже описано в
`ai-clone/feedback/format-prompt-matches-parse-mode.md`). Разведка (Explore-агент)
прошла по всем 9 агентам и `base_agent.py`, нашла и подтвердила несколько
реальных багов в проде, отсортированных по серьёзности ниже.

Главный системный вывод разведки: **фикс `looks_like_html()` в `_notify_user()`
защищает только путь «очередь задач → worker loop»**. Путь «живой диалог»
(`BaseAgent.handle_message()`) остаётся полностью незащищённым — если системный
промпт агента требует HTML (Тина, Макс), а Клод отвечает в свободном чате
(не через известную команду), результат уходит через `_clean_output()` (стирает
теги) → `_send_rich()` (GFM) — воспроизводится тот же баг, что был у Макса.

- [x] Строки ~513-537: сейчас `answer = _clean_output(answer); await _send_rich(...)`
  безусловно. Применить ту же развилку, что уже в `_notify_user()`
  (строки ~800-823): если `looks_like_html(answer)` — слать через
  `send_html_message()` напрямую (без `_clean_output`), иначе — текущий путь.
  Вынести общую логику «отправить текст агента с правильным форматом» в
  один приватный хелпер (например `_send_agent_text(chat_id, text, *, reply_to=None, reply_markup=None)`),
  используемый и в `_notify_user`, и в `handle_message` — избежать дублирования
  одной и той же развилки в двух местах.
- [x] Это чинит Тину (`TINA_SYSTEM` требует HTML на все ответы) и Макс
  (`_HELP_TEXT` с `<b>` встроен дословно в `MAX_SYSTEM`) в свободном диалоге —
  оба сейчас используют дефолтную `BaseAgent.handle_message` без переопределения.

## Фаза 2 — `agents/marta.py`: три «теневых» reply() в обход `_send_rich`

- [x] `handle_voice()` (~917-918): локальный `reply()` делает
  `update.message.reply_text(text, parse_mode=parse_mode)` с `parse_mode`,
  который вызывающий код никогда не передаёт (всегда `None`) — Rich Markdown
  (`**bold**`, `` `code` ``) от Клода уходит как есть, пользователь видит
  сырые звёздочки. Заменить на вызов `_send_rich`/`send_rich_or_fallback`
  (с той же `looks_like_html`-развилкой, что и в base_agent, для единообразия).
- [x] `handle_photo()` (~1118-1119) — та же проблема, тот же фикс.
- [x] `_handle_image_action()` (~1169-1170) — `context.bot.send_message(..., parse_mode=parse_mode)`,
  тот же паттерн, тот же фикс.
- [x] Fallback-путь при недоступном `task_queue` (~740-749): `header + _clean_output(result)`
  агента (может быть HTML-таблица Макса) уходит через GFM `_send_rich` —
  применить ту же `looks_like_html`-развилку, что в `_notify_user`.

Реализовано через уже существующий `self._send_agent_text()` (из Фазы 1).
Осознанно не тронуто (тот же класс проблемы, но вне заявленных трёх мест):
`reply()` в `handle_callback_query` (`chain_cancel`, ~строка 462) — уже
вызывает `_send_rich` напрямую, но без `looks_like_html`-развилки.

## Фаза 3 — `agents/kevin.py::_handle_gate_callback`

- [x] Строки ~363-364: `self.app.bot.send_message(chat_id, final_text or "Готово.")` —
  без `parse_mode`, без `_send_rich`. `final_text` (Rich Markdown после
  подтверждения PR) уходит как plain text. Завести через `_send_rich`/`send_rich_or_fallback`.
  Заодно точечный фикс `agents/peter.py:2702` — одиночная `*звёздочка*` (курсив)
  → `**двойная**` (жирный, как задумано).

## Фаза 4 — `agents/elina.py`: экранирование перед вставкой в HTML

- [x] `_extract_description_from_seo`/`_extract_title_from_seo` (~270-304) корректно
  вырезают `**`/`*`, но не делают `html.escape()` перед вставкой результата в
  HTML-превью (~383-396, `parse_mode="HTML"`). Если Клод вернёт в названии/описании
  символы `<`, `>`, `&` — сообщение сломается или отрендерится не так. Добавить
  `html.escape()` (см. `utils/tg_format.py::escape`) на извлечённый текст перед
  вставкой в HTML-шаблон.

## Фаза 5 — `agents/eva.py::_split_digest`

- [x] Строки ~122-147: эвристика искала `##`-заголовки как безопасные точки разреза
  чанков дайджеста, но `_DIGEST_PROMPT`/`_EMAIL_DIGEST_PROMPT` требуют HTML (не
  markdown-заголовки) — эвристика никогда не срабатывала, всегда использовался
  жёсткий разрез по длине, который мог разорвать `<blockquote expandable>`
  посередине и сломать HTML в одном из чанков. Заменено на HTML-осведомлённую
  логику: разрез по границам `\n\n`, но только там, где число открытых
  `<blockquote...>` равно числу `</blockquote>` (не рвёт тег); fallback —
  жёсткий разрез с `logger.warning`, если во всём блоке нет сбалансированной
  границы (например один `<blockquote>` длиннее лимита).

## Не берём в эту итерацию (осознанно)

- `agents/peter.py` — три разных формата в разных хардкод-командах
  (`parse_mode="HTML"` / legacy `"Markdown"` / GFM `_send_rich`) — каждый
  самосогласован (контент и способ отправки совпадают), корректность не
  нарушена. Это стилистический долг, не баг — трогать не будем, чтобы не
  рисковать рабочим кодом ради косметики. Единственная точечная правка:
  строка ~2702 `f"📸 *Нужна новая инфографика*..."` — одиночная звёздочка в
  Rich Markdown это курсив, а задумывался жирный (`**`) — тривиальный
  однострочный фикс, можно поправить заодно в Фазе 3 или отдельным мелким коммитом.
- `_format_chain_completion_message` (base_agent.py, эксцерпт 200 симв. на
  агента в финале цепочки) — теги уже стёрты, риск редкий и косметический
  (смазанный фрагмент таблицы в превью), не самостоятельная проблема формата
  доставки — не трогаем.
- `utils/tg_format.py` — `MARKDOWN_FORMAT_RULES`/`HTML_FORMAT_RULES`/`escape_mdv2`/`strip_mdv2`
  похожи на мёртвый код (никем не импортируются) — перед удалением обязательно
  перепроверить grep'ом на использование во всех агентах и тестах.
