# Рефакторинг антипаттернов

Статус: завершён (2026-07-19)

## Контекст

Выявлено 6 антипаттернов в ходе аудита кодовой базы. Два из них дёшевы в исправлении и дают высокую отдачу — их закрываем в первую очередь. Остальные — по мере касания файлов.

---

## Фаза 1 — Единый реестр агентов (приоритет: высокий) [x]

**Проблема:** словари `{agent_key → name/emoji}` продублированы в 6+ местах:
`base_agent.py`, `marta.py` (×2), `tools/tg_status.py`, `tools/notion.py`, и внутри `_advance_chain`.
Добавление нового агента требует обновления всех мест вручную.

При реализации (2026-07-19) выяснилось, что реальный масштаб больше описанного: 12+ инлайн-копий
в `marta.py` (не 2), а `tools/tg_status.py`/`tools/notion.py` из плана в репозитории не существуют
(план устарел на этот счёт, эти файлы никогда не создавались в таком виде).

- [x] Создать `agents/registry.py` с единым реестром (эмодзи Тины унифицировано на `🏛️` —
  в старом коде было расхождение `📋`/`🏛️` между разными местами, взят вариант большинства)
- [x] Заменить дубли в `base_agent.py` (`_AGENT_NAMES`, `_CHAIN_AGENT_EMOJI`, `_CHAIN_AGENT_NAME`)
  на производные от реестра
- [x] Заменить все 12 инлайн-словарей в `marta.py` на производные от реестра
- [x] `tools/tg_status.py`/`tools/notion.py` — не существуют, пункт снят как неактуальный

## Фаза 2 — Таймаут агента из реестра (приоритет: высокий) [x]

**Проблема:** `timeout_seconds=600 if next_agent == "dan" else 300` хардкодом.
Хрупко: переименование агента или добавление нового медленного сломает тихо.

При реализации выяснилось, что в текущем коде это вхождение ровно одно (не 4, как в плане —
остальные 3 адреса из плана устарели, там сейчас другой код).

- [x] Добавить в `agents/registry.py` поле `timeout` (уже заложено в Фазе 1)
- [x] Заменить `600 if next_agent == "dan" else 300` (`base_agent.py`) на `agent_timeout(next_agent)`

## Фаза 3 — Убрать `pool=None` из сигнатур (приоритет: средний) [x]

**Проблема:** `get_chain_plan(None, chain_id)` и `enqueue_chain_task(pool=None, ...)` — параметр `pool` врёт: `None` это нормальный рабочий путь, а не «не передан».

- [x] В `task_queue.py`: убран параметр `pool` из `get_chain_plan`, `enqueue_chain_task`,
  `get_chain_results` — тела и так вызывали `get_pool()` сами. Заодно починена `get_chain_status`
  (тот же антипаттерн, не была в списке плана, но не вызывается нигде в репо). `count_incomplete_in_group`
  не трогалась — у неё уже не было параметра `pool` (план был неточен на этот счёт).
- [x] Обновлены все вызывающие места в `base_agent.py` (5 мест) и `agents/marta.py` (1 место).

## Фаза 4 — Напоминания Алекса в BaseAgent (приоритет: низкий) [x]

**Проблема:** `base_agent.py:690–714` — каждый агент (Марта, Питер, Макс…) при каждой 30-й итерации делает запрос к БД за напоминаниями. Это логика Алекса.

- [x] Общий блок (`cleanup_timed_out_tasks` + уведомление о зависших цепочках — не Alex-специфично,
  оставлено общим) вынесен в новый переопределяемый метод `BaseAgent._periodic_maintenance()`.
  `AlexAgent._periodic_maintenance()` вызывает `super()._periodic_maintenance()`, затем добавляет
  проверку `get_due_reminders`/`send_push` — теперь этим занимается только Алекс, а не каждый агент.
